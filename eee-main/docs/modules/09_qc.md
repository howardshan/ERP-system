## 概述

宠物零食制造业的"**烘干后 QC 闭环**"模块,核心解决[QC模块起步与设计指南](../../../docs/QC模块起步与设计指南.md) §1.4 描述的:
> 中午烘干完、傍晚才收到结果 → 不合格品堆积。

闭环流程:
```
生产批 → 进烘干房(check_in)
       → 出烘干房(check_out → pending)
       → QC 员录入 Aw(submit_inspection → passed / hold)
       → 管理员处置 Hold(create_disposition → closed)
```

模块用同一套子批模型 + 状态机 + 事件流,覆盖待检看板、批次追溯、Hold 处置、SKU/模板维护等场景。

## Production 临时创建页

`Production` 页面是 QC 模块内的**临时创建向导**(因为还没有独立的 Production 模块)。一次提交同时创建 1 个 Dry Room(production_lot) + N 个 Sub-lot。

**字段**(顶部 production header):
- production_date(默认今天)
- shift(early / late / night / other)
- product SKU
- work_order_barcode
- production_code(默认按 `P-{YYYYMMDD}-{SHIFT}-{SKU}` 自动生成,可手动覆盖)

**Sub-lot 列表**(下半部分):
- 每行只需要 Sub-lot 编号(留空则自动生成 `<production_code>-D##`)
- "Add sub-lot" 可加多行,"Remove" 按钮删行
- 烘干位置和 check-in 时间**不在此处录入** —— 实际入烘干房时再录(在 Dry Room 详情页 check-in)

**实现**:`createProductionBatch` (qcApi.ts) 在前端层串行调用 `createProductionLot` + 多次 `qc_check_in_sub_lot` RPC,没有引入新表。Sub-lot 创建出来的初始 status 是 `drying`,location_id 为 NULL,in_time 默认 `now()`。

### 打印 cart stickers(2026-05-25 起,M-084 浏览器打印路径;2026-06-10 起 M-121 改 4×3 + 一键打印助手;M-122 改矢量 PDF)

成功创建 work order、Batch Trace 加车 / 重打贴纸时,会打开 [`CartStickerSheet`](../../src/pages/qc/components/CartStickerSheet.tsx) 模态:每车一张 **4×3 inch 横版**标签(真实尺寸,4 inch 宽 × 3 inch 高)。依赖 [jsbarcode](https://github.com/lindell/JsBarcode) 渲染 Code 128 条码。

> **M-122(2026-06-10)** 为解决位图标签字边发毛:Web 打印路径由"canvas PNG 位图"改为 **jsPDF 生成的矢量 PDF**。文字/线条保持矢量,**由打印机驱动按原生分辨率现场栅格化**,任意型号都锐(无固定分辨率位图、无锯齿);车号码改用 **QR 二维码**(`qrcode` 库,编码完整 `sub_lot_code`):一维 CODE128 的窄条在热敏上受**点扩张**影响、条宽比例失真扫不出;QR 用大模块+纠错,对热渗不敏感、不挑朝向,小方块即可。按打印机原生 dpi 画成整数对齐的纯 1-bit 方块、`addImage` 1:1 嵌入(放左列,下方保留 CART#+号人眼可读)。**需二维影像式扫码枪**(一维激光读不了 QR)。预览改为直接内嵌该 PDF(`<iframe>`,所见即所打)。桥新增 `{ pdf }` 入口:macOS `lp file.pdf`、Windows 用**同目录 SumatraPDF.exe** `-silent -print-settings noscale` 静默打印(便携版放 `scripts/print-bridge/vendor/SumatraPDF.exe`,由 `build.mjs` 打进 Windows 包、安装脚本拷到运行目录)。dpi 选择器仍保留(仅 Tauri PNG 路径用得到)。

> 1. 标签尺寸由 2×3(设计 6×4)改为真实 **4×3 横版**,布局按真实毫米(101.6 × 76.2 mm)重排。

## Testing 简化流程(用户反馈调整)

原本的 "Preview judgment" 中间步骤已经去掉。新流程(M-109 起改为人工裁定):

1. 取样:输 sample_id → 点 `取样`
2. Measure:数字键盘录 WA → 系统按 SKU 模板给出**建议**("System suggests PASS/FAIL",仅参考)
3. **Final judgment**:操作员用 Pass / Fail 两个按钮拍板,默认跟随系统建议、**可覆盖**(覆盖时读数框旁标 "overridden",并记 `manual_override`);水活之外的综合判断由人工决定
4. **Remark**(选填):文本框记录裁定备注,经 `qc_inspection_record.remark` 持久化,Full History 抽屉可调取
5. 底部 **Redo test**(清空读数,sample 不变)/ **Confirm PASS/FAIL & persist**(颜色随操作员决定变,未选时灰态)

Pass → 进 `done` step(绿色 PASS · released);Fail → 弹 DispositionPicker。无 SKU 模板时无建议,但仍可人工裁定(BR-Q67)。

**头部时间信息(M-108)**:Testing 详情头部卡片在 SKU 名下方显示一行 `Produced <时间> · Drying done <时间>`。`Produced` = `produced_at`(即 `qc_production_lot.created_at`,工单建立/入系统时间,schema 无专门生产完成时间戳的代理);`Drying done` = `out_time`(出烘干房 / check-out 时间)。

## Sampling Groups(M-048,批量出库 + champion 抽样;M-118 改为确定性 + 操作员选方法)

为减少 Testing 工作量,出 dryer 不再 1:1 测试,而是按 SKU 上配置的 `sample_every_n_carts` 抽样:每 N 车选 1 车作为 **champion**,其余车进 `awaiting_group_result` 等结果。**M-118 起 champion 不再随机,改为确定性,且操作员每次出库可选两种切片方式之一**。

**配置**:`qc_product_sku.sample_every_n_carts`(默认 1 = 全检),在 Product 模板表单里可改。

**触发**:`DryRoomListMode` 右侧多选 drying carts → "Check out (N)" → `BulkCheckOutDialog` 显示 work order 分组预览 + 方法 radio → 后端 `qc_check_out_sub_lots_bulk` RPC 一次性出库 + 拆组。

**拆组规则(BR-Q27,经 M-118 修订)**:同一次 RPC 调用内,按 `production_lot_id`(redry 时还按 `original test_group_id`)分桶,每桶里 `sub_lot_code` **降序**(高号车在前)排列。操作员在弹窗选择切片方式:

- **Method 1**(默认,按 N 切片):每 N 车一组,余数不足 N 自成一组;每组 champion = 该组**最大编号**(降序首位)。例 T=10/N=3 → 4 组,champion = 10,7,4,1。
- **Method 2**(余数并入末组):若 `R = T mod N > 0` 且 `T > N`,前 `⌊T/N⌋ − 1` 组常规(N 车,champion = 组内最大),末组 `N + R` 车,champion = 升序排第 `⌊K/2⌋ + 1` 位(**中偏大**)。例 T=10/N=3 → 3 组,champion = 10,7,3。R = 0 或 T ≤ N 时两种方法等价。
- 每个 `group_assigned` 事件 payload 与 RPC 返回的 `groups[*]` 都带上 `sampling_method`,审计/分析可还原。
- Step 2b(redry 重组)与 Step 2a 用同一 `p_sampling_method`;每次新的批量出库**重新选**,不持久化沿用上次(决议)。
- 前端预览与 SQL 共用 [`src/lib/qcSampling.ts`](../../src/lib/qcSampling.ts) 的 `planSamplingGroups()`,改算法必须**两边同步**。详见 [`docs/QC-取样分组方式-修改计划.md`](../QC-取样分组方式-修改计划.md)。

**BR-Q68(M-118)**:批量出烘干时,操作员必须选择抽样方式(默认 Method 1)。建组按 `sub_lot_code` 降序进行;Method 1 = 每 N 辆切一组、余数自成一组、champion = 组内最大;Method 2 = 若有余数则少做 1 组常规组、把余数并入末组、末组 champion = 升序第 `⌊K/2⌋+1` 位(中偏大);常规组始终取最大。同一次 bulk 调用 Step 2b(redry)沿用同一方法参数。**前提**:`sub_lot_code` 末位为定长数字 seq,词典序与数字序一致(若 seq 格式变化需重新审视降序语义)。

**Testing 阶段联动(BR-Q28,经 M-055 / M-106 修订)**:
- Champion PASS → 同组所有兄弟车升 `passed`(自动放行,无需操作)
- Champion FAIL → champion **及全部兄弟车**一起转 `hold`(M-055:它们共担 champion 命运),group 标 `closed_failed`;之后按车从 QC Home 处置
- **retest 是组级动作(M-106)**:对组内任意车选 `retest` → 被 retest 的车成为唯一 champion(`pending`),同组其余仍在测试态的车回 `awaiting_group_result`,group 重开(`sampling`)。前端 `createDispositionGroup` 对 retest **只调一次**;一次重测结果即覆盖全组。**默认保留原 champion**:`createDispositionGroup` 接受 `champion_sub_lot_id`(QcHome 传 `item.drying_sub_lot_id`,即「Needs attention」卡片对应的那辆被测车),retest 就发在它身上,避免 M-106 把 champion 漂移到 `sub_lot_ids[0]`(升序首位、最低号车)那种和操作员直觉不符的情况。solo 车或老调用方不传该参数时回退到 `sub_lot_ids[0]`
- Champion 选 `redry_dryer` / `room_temp_dry` / `scrap` 等其他 disposition → 逐车独立处理(失败组每辆车各自重烘 / 室温干 / 报废;重烘车再次批量出库时按 Step 2b 重新分组)
- **传播口径(M-106)**:`qc_submit_inspection` 把同组「非 champion 且处于 `awaiting_group_result` 或 `pending`」的兄弟车一并改判,作为防御兜底,避免任何残留 pending 兄弟漏接结果
- **Needs attention 去重(M-107)**:QC Home「Needs attention」按组去重,每个抽样组只显示一张卡(取该组当日最新一次检测),solo 车每车一张。避免同组多次 retest 失败叠出多张重复卡片(BR-Q66)
- **Failed today 拆分(M-120)**:Dashboard「Failed today」**总数仍是今天所有失败次数**(不会因 retest 通过而下降);卡片大数字下方多一行小字 `<n> still open · <m> resolved`,把还没消化的子集单独显示,以分清"今天踩了多少坑"和"还有多少坑要填"。Fail 明细面板每行右上角加 outcome chip(`Retested → passed` / `Disposed` / `Still open`)。判定口径:同卡或同抽样组后续事件;优先级 `retest_passed` > `disposed` > `open`(BR-Q69)。**M-140 起 `disposed` = 后续存在任意类型处置**(retest / redry / room_temp / scrap / grind / concession / rework),与 Needs-Attention 列表的口径一致 —— 操作员一旦做了处置,该 fail 即不再计入「仍待处理」(原 M-120 只认 scrap/grind/concession/rework 终态处置,导致 retest 后仍显示待处理)。retest 若再次失败会生成新的 fail 行,重新计为 open。
- **Passed today 明细面板(M-121)**:`Passed today` 卡片可点击展开最近 2 天的 pass 明细(镜像 Failed today 的体验),每行 chip 标 `Released`(同组所有成员已脱离 `passed`)或 `Awaiting release`(至少一辆还在 `passed`);solo 车按自己的 `status` 判定(BR-Q70)

**UI 反馈**:`TestingPage` 在 pending queue 给 champion 显示 `Users ×N` 紫色 badge;选中后页头出现紫色 banner 解释"PASS 整组放行 / FAIL 仅 champion / Retest re-roll"。

**新状态**:`qc_drying_sub_lot.status` 加 `awaiting_group_result`。STATUS_LABEL/COLOR 用紫色家族。

## Days vs Minutes(M-050,UI / DB 单位约定)

- **BR-Q45** List mode 的扫描错误(不合资格 / 状态不对)用**居中弹窗 + OK 按钮**显示,不再是顶部红色横条——长时间扫码场景下,操作员不容易错过弹窗;一键 OK / 点 backdrop / 点 ✕ 都能关闭。

## 状态机

```
(null) ── create_sub_lot ──→ created          -- Production 创建,不在烘干房
created ── register_in_dryer ──→ drying       -- Dry Room 详情页 awaiting check-in
(null) ── register_in ──→ drying              -- LotDetail 直接 check-in(legacy 路径,跳过 created)
drying ── move (target empty) ──→ drying      -- 自己换格子
drying ── move (target occupied)
       → 自己进新格 (drying),被挤的进 awaiting_recheck
awaiting_recheck ── register_in_dryer ──→ drying  -- 重新放回任意空格,累计时长续算
drying ── register_out ──→ pending                                   -- 单车出库(legacy/小批)
drying ── check_out_bulk (champion) ──→ pending                      -- 批量出库 + 同组挑中的 champion
drying ── check_out_bulk (sibling)  ──→ awaiting_group_result        -- 批量出库 + 同组未中 champion
pending ── take_sample ──→ pending          -- 取样不改状态,创建 sample 记录
pending ── submit_inspection (pass) ──→ passed
                                          ── (champion) ──→ 同组兄弟(awaiting_group_result / 非champion pending)→ passed (BR-Q28, M-106)
pending ── submit_inspection (fail) ──→ hold
                                          ── (champion) ──→ 同组兄弟一并 → hold,组标 closed_failed (M-055)
hold ── disposition (retest) ──→ pending(被 retest 的车成为唯一 champion)
                              + 同组其余测试态兄弟 → awaiting_group_result,组重开 sampling (M-106)
hold ── disposition (redry_dryer) ──→ awaiting_recheck  -- 新 expected_dry_minutes
hold ── disposition (room_temp_dry) ──→ room_temp_drying (open session)
hold ── disposition (scrap/concession/...) ──→ closed
room_temp_drying ── stop_room_temp_dry ──→ pending  -- 回到 Testing 队列重新取样测
pending ── start_inspection ──→ inspecting
inspecting ── submit_pass ──→ passed
inspecting ── submit_fail ──→ hold
hold ── start_disposition ──→ disposing
disposing ── complete_disposition ──→ closed
closed ── pkg_dispatch_carts ──→ dispatched  -- (M-067) Packaging 打包出库,终态
```

`qc_drying_sub_lot.status` 的 CHECK 约束限制了取值集合;转移逻辑则强制在 SQL RPC 函数里(`qc_check_in_sub_lot` / `qc_check_out_sub_lot` / `qc_submit_inspection` / `qc_create_disposition` / `pkg_dispatch_carts`)。前端**不**自行做状态转移,只调 RPC。

事件流写入 `qc_quality_event`,事件类型:`check_in`、`check_out`、`inspection_passed`、`inspection_failed_hold`、`disposition_completed`。

**业务规则**:
- **BR-Q35** 每个 QC 页面必须有对应的页面级 view 权限检查;不通过时 **不调用 RPC,不渲染数据**,而是显示 `PermissionDenied` 面板说明所缺权限。
- **BR-Q36** 动作按钮(Release / Dispose / Sample / Submit / Check-in / Check-out / Move / Stop room-temp / Dispatch / 编辑 SKU 等)的 `disabled` 必须独立于页面 view 检查,使用对应的 action permission(已有的 `canRelease` / `canSample` / `canCheckOut` 等模式)。这样用户能看页面但不能动 = 只读授权;不能看页面 = 无权限。

**Hooks 顺序提醒**: `if (!canView) return <PermissionDenied .../>` 必须放在**所有 useState / useEffect / useMemo 之后**才能 early-return,否则违反 React Rules of Hooks。`PermissionDenied` 组件本身不带 hooks,可在任何位置安全使用。

## QC Home 功能

`QcHome`([QcHome.tsx](../../src/pages/qc/QcHome.tsx))是 QC 模块的默认首页,整合了仪表盘与操作入口。

### Stat Cards(统计卡片)

顶部一行卡片,数据来自 `qc_overview()` RPC:

| 卡片 | 说明 |
|------|------|
| Currently Drying | 当前 `drying` 状态的车数 |
| Awaiting Sample | `pending` 且无 pending sample 的车数 |
| Awaiting WA Result | `pending` 且有 pending sample 或 `inspecting` 的车数 |
| Room Temp Drying | 当前 `room_temp_drying` 状态的车数 |
| Passed Today | 今日 `passed` 结果数(绿色卡片) |
| Failed Today | 今日 `failed` 结果数(红色卡片);点击展开 **FAIL Detail Panel** |

**FAIL Detail Panel**(`FailDetailPanel`):点击 Failed Today 卡片展开,调用 `qc_recent_failed_inspections(p_days=2)` RPC,列出近 2 天的失败检验记录及其组成员信息(M-059)。

### Needs Attention(需关注列表)

- **每行一个 sample**,而非每组一行(M-057 后)
- 每行显示:sub-lot 代码、AW 值、判定结果(PASS/FAIL)、当前状态、工单编号
- PASS 行:展示 Release 按钮 → 打开 `ReleaseDialog` → 确认即放行 → 调 `qc_release_passed_sub_lot(subLotId)`(**M-139 起不再录入 yield**),完成后行自动消失。**放行不过账 ERP 数量**(产出数量留待后续环节)。历史卡片 packaging_item_id 为 NULL 时,Dialog 会按 SKU 关联数自动 fill / 弹包装选择器 / 报错(决议 §5.7,M-114)
- FAIL 行:展示 Dispose 按钮 → 打开 `DisposeDialog`(M-046 组件),完成后行自动消失
- **Group badges**:若有 `test_group_id`,额外显示同组其他车的代码 badge(group_sub_lot_codes)
- **自动清理**(M-065/M-066):只显示仍需人工干预的条目 — FAIL 需组内仍有 `hold`/`awaiting_group_result` 车;PASS 需组内仍有 `passed` 车(等待 release)

### Released Inventory(已释放库存图表)

QC Home 底部新增板块(M-067),数据来自 `pkg_inventory_summary()` RPC:

- 每个 SKU 一行,横向堆叠色条展示在库车辆的天数分布
- 绿色:`<10` 天;琥珀色:`10-14` 天;红色:`≥15` 天
- 提示操作员哪些 SKU 积压时间过长,优先安排打包出库

## 测试结果邮件通知(M-083 / EF-004,Phase 1)

每次记录 QC 测试(`qc_submit_inspection` 写入 `qc_inspection_record`)后,自动给开启了该通知的用户发邮件。

- **触发**:`trg_qc_notify_on_inspection`(`AFTER INSERT ON qc_inspection_record`)→ pg_net 调 `send-notification` Edge Function(EF-004)。fire-and-forget,通知失败**不影响**测试结果落库(BR-N2)。
- **粒度**:按需求决策「每次测试都发」。组测试只有 champion 写 inspection record,故每次真实提交恰好一封,传播不重复触发。
- **邮件内容**:本批结果(sub-lot / SKU / lot / Aw / PASS-FAIL / inspector / sample)+ 今日汇总(合格、不合格、等待测试数、在烘数、合格率),后者复用 `qc_overview()->'stats'`。
- **抽样组覆盖(M-085)**:若被检测车属于多车抽样组,邮件额外显示「Sampling group — N carts covered」,列出全组 sub-lot 并高亮被抽检(代表)那一车。组测试只检 champion,结果适用全组(pass→全过,fail→全挂,见 M-048/M-056)。payload 字段 `is_group` / `group_size` / `group_members[{code,tested}]`。
- **收件人**:由 `notification_recipients('qc_test_result')` 按三态生效规则解析(见 06_users-auth.md 与 M-083)。Phase 1 默认 `tianzuohuang@crave-cook.com`。
- **配置**:见 `docs/database/03_migrations-and-edge-functions.md` 的 M-083 / M-084 / EF-004(SMTP2Go secret;共享暗号经 Supabase Vault `notify_webhook_secret` 提供——M-084)。
- **Phase 2**:管理员侧通知设置 UI(按模块分卡片)+ 账户设置页自助开关。
