## 概述

宠物零食制造业的"**烘干后 QC 闭环**"模块,核心解决[QC模块起步与设计指南](../../../docs/QC模块起步与设计指南.md) §1.4 描述的:
> 中午烘干完、傍晚才收到结果 → 不合格品堆积。

闭环流程:
```
生产批 → 进烘干房(check_in)
       → 出烘干房(check_out → pending)
       → QC 员录入 Aw(submit_inspection → passed / hold)
       → 管理员处置 Hold(create_disposition → closed)
```

模块用同一套子批模型 + 状态机 + 事件流,覆盖待检看板、批次追溯、Hold 处置、SKU/模板维护等场景。

## Production 临时创建页

`Production` 页面是 QC 模块内的**临时创建向导**(因为还没有独立的 Production 模块)。一次提交同时创建 1 个 Dry Room(production_lot) + N 个 Sub-lot。

**字段**(顶部 production header):
- production_date(默认今天)
- shift(early / late / night / other)
- product SKU
- work_order_barcode
- production_code(默认按 `P-{YYYYMMDD}-{SHIFT}-{SKU}` 自动生成,可手动覆盖)

**Sub-lot 列表**(下半部分):
- 每行只需要 Sub-lot 编号(留空则自动生成 `<production_code>-D##`)
- "Add sub-lot" 可加多行,"Remove" 按钮删行
- 烘干位置和 check-in 时间**不在此处录入** —— 实际入烘干房时再录(在 Dry Room 详情页 check-in)

**实现**:`createProductionBatch` (qcApi.ts) 在前端层串行调用 `createProductionLot` + 多次 `qc_check_in_sub_lot` RPC,没有引入新表。Sub-lot 创建出来的初始 status 是 `drying`,location_id 为 NULL,in_time 默认 `now()`。

成功创建 work order 或往现有 work order 加车后,绿色 success 横条右侧会出现 **"Print N stickers"** 按钮。点开打开 [`CartStickerSheet`](../../src/pages/qc/components/CartStickerSheet.tsx) 模态:每车一张 4×6 inch 贴纸,浏览器原生 print dialog 出图。

## Testing 简化流程(用户反馈调整)

原本的 "Preview judgment" 中间步骤已经去掉。新流程:

1. 取样:输 sample_id → 点 `取样`
2. Measure:数字键盘录 WA → 顶部读数框**实时**显示绿/红 + "Within spec — Pass" / "Out of spec — Fail" 文案
3. 底部固定两个按钮:
   - **Redo test** — 清空当前 WA 读数(sample 不变,可再录一次)
   - **Confirm PASS/FAIL & persist** — 颜色随 judged 结果变(绿/红),WA 未录时为 "Enter a reading first" 灰态

Pass → 进 `done` step(绿色 PASS · released);Fail → 弹 DispositionPicker。

## Sampling Groups(M-048,批量出库 + champion 抽样)

为减少 Testing 工作量,出 dryer 不再 1:1 测试,而是按 SKU 上配置的 `sample_every_n_carts` 抽样:每 N 车随机选 1 车作为 **champion**,其余车进 `awaiting_group_result` 等结果。

**配置**:`qc_product_sku.sample_every_n_carts`(默认 1 = 全检),在 Product 模板表单里可改。

**触发**:`DryRoomListMode` 右侧多选 drying carts → "Check out (N)" → `BulkCheckOutDialog` 显示 work order 分组预览 → 后端 `qc_check_out_sub_lots_bulk` RPC 一次性出库 + 拆组。

**拆组规则(BR-Q27)**:同一次 RPC 调用内,按 `production_lot_id` 分桶,每桶按 SKU 的 `sample_every_n_carts` 切片,每片形成一个 `qc_test_group`,随机挑 1 辆 `is_test_champion = true` 进 `pending`,其余进 `awaiting_group_result`。最后一片不足 N 也单独成组(roundup)。

**Testing 阶段联动(BR-Q28)**:
- Champion PASS(`qc_submit_inspection` 返回 pass)→ 同组所有 `awaiting_group_result` 兄弟车 batch 升 `passed`(自动放行,无需操作)
- Champion FAIL → 仅 champion 转 `hold`,兄弟车继续 `awaiting_group_result` 等下一轮
- Champion 在 hold 下选 `retest` disposition → 从同组兄弟车随机抽 1 辆晋升为新 champion(`awaiting_group_result` → `pending`、`is_test_champion = true`),旧 champion 进 `closed`,组 ID 不变,member_count 递减 1
- Champion 选 `redry_dryer` / `room_temp_dry` / `scrap` 等其他 disposition → 仅处理 champion,兄弟车按现状保留(可后续人工调整,Phase 2 加批量处理)

**UI 反馈**:`TestingPage` 在 pending queue 给 champion 显示 `Users ×N` 紫色 badge;选中后页头出现紫色 banner 解释"PASS 整组放行 / FAIL 仅 champion / Retest re-roll"。

**新状态**:`qc_drying_sub_lot.status` 加 `awaiting_group_result`。STATUS_LABEL/COLOR 用紫色家族。

## Days vs Minutes(M-050,UI / DB 单位约定)

干燥时间在 UI 上全部以"天"为主单位渲染(`fmtDays` / `fmtDuration`),DB 列名/值保持 `*_minutes`(分钟)以兼容已有的 `expected_finish_at` 倒计时逻辑。换算用 `MINUTES_PER_DAY = 1440` 常量,辅助函数:`minutesToDays(min)` / `daysToMinutes(days)`(均在 `src/lib/utils.ts`)。

Production 表单、ProductManagement、LotDetail header 的输入框都换成"days"(支持小数),提交时转分钟入库。已有 sub-lot 上的 `expected_dry_minutes` 不需要迁移 — 显示时通过 helper 自动以天/小时混合形式呈现(如 1d 6h / 18h / 45m)。

## Duplicate-scan warning(M-050,BR-Q34)

`DryRoomListMode` 的两个 scan flow(check-in 选择 / check-out 选择)各自维护 session Set 记下已扫 sub-lot_code。再扫同 code 时:

session Set 在 bulk check-in/out 成功后 reset。Scan flow 不影响人工 checkbox 勾选。

## Pass-rate forecast(M-050,QcHome 卡片网格)

## Check-in to Dryer 网格页

[`CheckInToDryer.tsx`](../../src/pages/qc/CheckInToDryer.tsx) 是 Production 之后的物理放置环节。

- 左栏:状态为 `created` 的 sub-lot 列表(等待入烘干房)
- 右栏:5 个 dryer 标签,每个 10×10 网格,格子编号 **00 ~ 99**(行优先,`row * 10 + col`)
- 颜色:灰 = 未选 sub-lot,绿 = 可用,蓝 = 选中,红 = 已被其他 active sub-lot 占用
- 操作:点击 sub-lot → 点击格子 → "Confirm check-in" → 后端 `qc_register_in_dryer` 把状态从 `created` 切到 `drying`,写入 `location_id` + `in_time` + `check_in` 事件
- 占用判定:`status ∈ (drying / pending / inspecting / hold / disposing)` 的 sub-lot 锁住格子;`passed` / `closed` / `created` 不锁

## 状态机

```
(null) ── create_sub_lot ──→ created          -- Production 创建,不在烘干房
created ── register_in_dryer ──→ drying       -- Dry Room 详情页 awaiting check-in
(null) ── register_in ──→ drying              -- LotDetail 直接 check-in(legacy 路径,跳过 created)
drying ── move (target empty) ──→ drying      -- 自己换格子
drying ── move (target occupied)
       → 自己进新格 (drying),被挤的进 awaiting_recheck
awaiting_recheck ── register_in_dryer ──→ drying  -- 重新放回任意空格,累计时长续算
drying ── register_out ──→ pending                                   -- 单车出库(legacy/小批)
drying ── check_out_bulk (champion) ──→ pending                      -- 批量出库 + 同组挑中的 champion
drying ── check_out_bulk (sibling)  ──→ awaiting_group_result        -- 批量出库 + 同组未中 champion
pending ── take_sample ──→ pending          -- 取样不改状态,创建 sample 记录
pending ── submit_inspection (pass) ──→ passed
                                          ── (champion) ──→ siblings 同组 awaiting_group_result → passed (BR-Q28)
pending ── submit_inspection (fail) ──→ hold
hold ── disposition (retest, champion w/ siblings) ──→ closed (旧 champion)
                                                    + 同组随机 awaiting_group_result → pending (新 champion)
hold ── disposition (redry_dryer) ──→ awaiting_recheck  -- 新 expected_dry_minutes
hold ── disposition (room_temp_dry) ──→ room_temp_drying (open session)
hold ── disposition (retest) ──→ pending               -- 不重烘,回 Testing 取新 sample
hold ── disposition (scrap/concession/...) ──→ closed
room_temp_drying ── stop_room_temp_dry ──→ pending  -- 回到 Testing 队列重新取样测
pending ── start_inspection ──→ inspecting
inspecting ── submit_pass ──→ passed
inspecting ── submit_fail ──→ hold
hold ── start_disposition ──→ disposing
disposing ── complete_disposition ──→ closed
closed ── pkg_dispatch_carts ──→ dispatched  -- (M-067) Packaging 打包出库,终态
```

`qc_drying_sub_lot.status` 的 CHECK 约束限制了取值集合;转移逻辑则强制在 SQL RPC 函数里(`qc_check_in_sub_lot` / `qc_check_out_sub_lot` / `qc_submit_inspection` / `qc_create_disposition` / `pkg_dispatch_carts`)。前端**不**自行做状态转移,只调 RPC。

事件流写入 `qc_quality_event`,事件类型:`check_in`、`check_out`、`inspection_passed`、`inspection_failed_hold`、`disposition_completed`。

## 业务规则 (BR)

| 编号 | 内容 | 实现位置 |
|------|------|---------|
| **BR-Q1** | Aw 合格判定为闭区间 `[lower_limit, upper_limit]`,超出范围即 fail。 | `qc_submit_inspection` (M-034) |

### 页面级 view-permission 兜底(2026-05-23 起)

之前 QC 每个页面只在**动作按钮**上做 `can(...)` 判断,页面骨架本身在用户没有 view 权限时仍会渲染、调底层 RPC、暴露数据。本次把每个页面的顶层都加了 **view-permission gate**;不通过则渲染统一的 [`PermissionDenied`](../../src/pages/qc/components/PermissionDenied.tsx) 面板。

**业务规则**:
- **BR-Q35** 每个 QC 页面必须有对应的页面级 view 权限检查;不通过时 **不调用 RPC,不渲染数据**,而是显示 `PermissionDenied` 面板说明所缺权限。
- **BR-Q36** 动作按钮(Release / Dispose / Sample / Submit / Check-in / Check-out / Move / Stop room-temp / Dispatch / 编辑 SKU 等)的 `disabled` 必须独立于页面 view 检查,使用对应的 action permission(已有的 `canRelease` / `canSample` / `canCheckOut` 等模式)。这样用户能看页面但不能动 = 只读授权;不能看页面 = 无权限。

**Hooks 顺序提醒**: `if (!canView) return <PermissionDenied .../>` 必须放在**所有 useState / useEffect / useMemo 之后**才能 early-return,否则违反 React Rules of Hooks。`PermissionDenied` 组件本身不带 hooks,可在任何位置安全使用。

## QC Home 功能

`QcHome`([QcHome.tsx](../../src/pages/qc/QcHome.tsx))是 QC 模块的默认首页,整合了仪表盘与操作入口。

### Stat Cards(统计卡片)

顶部一行卡片,数据来自 `qc_overview()` RPC:

| 卡片 | 说明 |
|------|------|
| Currently Drying | 当前 `drying` 状态的车数 |
| Awaiting Sample | `pending` 且无 pending sample 的车数 |
| Awaiting WA Result | `pending` 且有 pending sample 或 `inspecting` 的车数 |
| Room Temp Drying | 当前 `room_temp_drying` 状态的车数 |
| Passed Today | 今日 `passed` 结果数(绿色卡片) |
| Failed Today | 今日 `failed` 结果数(红色卡片);点击展开 **FAIL Detail Panel** |

**FAIL Detail Panel**(`FailDetailPanel`):点击 Failed Today 卡片展开,调用 `qc_recent_failed_inspections(p_days=2)` RPC,列出近 2 天的失败检验记录及其组成员信息(M-059)。

### Needs Attention(需关注列表)

- **每行一个 sample**,而非每组一行(M-057 后)
- 每行显示:sub-lot 代码、AW 值、判定结果(PASS/FAIL)、当前状态、工单编号
- PASS 行:展示 Release 按钮 → 调 `qc_release_passed_sub_lot`,完成后行自动消失
- FAIL 行:展示 Dispose 按钮 → 打开 `DisposeDialog`(M-046 组件),完成后行自动消失
- **Group badges**:若有 `test_group_id`,额外显示同组其他车的代码 badge(group_sub_lot_codes)
- **自动清理**(M-065/M-066):只显示仍需人工干预的条目 — FAIL 需组内仍有 `hold`/`awaiting_group_result` 车;PASS 需组内仍有 `passed` 车(等待 release)

### Released Inventory(已释放库存图表)

QC Home 底部新增板块(M-067),数据来自 `pkg_inventory_summary()` RPC:

- 每个 SKU 一行,横向堆叠色条展示在库车辆的天数分布
- 绿色:`<10` 天;琥珀色:`10-14` 天;红色:`≥15` 天
- 提示操作员哪些 SKU 积压时间过长,优先安排打包出库

## 测试结果邮件通知(M-083 / EF-004,Phase 1)

每次记录 QC 测试(`qc_submit_inspection` 写入 `qc_inspection_record`)后,自动给开启了该通知的用户发邮件。

- **触发**:`trg_qc_notify_on_inspection`(`AFTER INSERT ON qc_inspection_record`)→ pg_net 调 `send-notification` Edge Function(EF-004)。fire-and-forget,通知失败**不影响**测试结果落库(BR-N2)。
- **粒度**:按需求决策「每次测试都发」。组测试只有 champion 写 inspection record,故每次真实提交恰好一封,传播不重复触发。
- **邮件内容**:本批结果(sub-lot / SKU / lot / Aw / PASS-FAIL / inspector / sample)+ 今日汇总(合格、不合格、等待测试数、在烘数、合格率),后者复用 `qc_overview()->'stats'`。
