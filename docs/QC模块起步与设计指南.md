## 1. 背景与问题定义

### 1.1 客户现状

- 已在使用一套 ERP，在 **财务、采购、销售** 等方面基本可用。
- **生产制造、车间、QC、批次追溯** 能力薄弱，满意度低。
- QC 环节以 **纸质记录为主**，事后由管理人员 **手工录入 ERP** 或汇总 Excel，导致：
  - 数据滞后（T+0 现场 → 数小时后甚至更晚才到管理层，详见 [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议））
  - 库存/放行状态与现场真实质量不一致
  - 追溯、客诉、召回响应慢
  - 管理看板无法反映「当前产线真实状态」
  - 不合格品因处置决策滞后而 **现场堆积**（烘干房、产线已投入后续任务）

### 1.2 行业特征（宠物零食 / 食品级制造）


| 特征    | 对系统的要求               |
| ----- | -------------------- |
| 强批次管理 | 原料批、生产批、成品批全链路关联     |
| 强 QC  | 来料 / 过程 / 成品多节点检验    |
| 强追溯   | 由成品批反查原料批、产线、人员、时段   |
| 强合规   | 检验记录留存、审计追踪、不合格品处置留痕 |
| 现场环境  | 手套、油污、少打字、多扫码/点选     |


### 1.3 核心问题（一句话）

> **生产现场质量数据没有实时数字化**，而不是「缺少一套新财务系统」。

### 1.4 客户现场痛点：烘干后 QC 延时（第一次会议）

本节记录与客户管理人员第一次会议中确认的 **首要痛点场景**。Phase 1 MVP 应优先覆盖此链路，其它 QC 节点（来料检、成品包装检等）可在同架构上扩展。

#### 1.4.1 现场流程描述（现状）

```text
产线生产完成
  → 同一生产批的产品分批送入烘干房
  → 烘干完成（各子批出房时间可能不同）
  → QC 员用仪器逐批检测（仪器无法接入系统）
  → 人工读数 + 纸笔记录
  → 在对应批次上贴「合格 / 不合格」纸质标签
  → 管理人员将数据与结果汇总至 Excel
  → 提交上层管理决策
  → 处置：返烘 / 粉碎后重新投入产线 / 其它方式
```

**工艺与质量变异（次品来源）：** 同一生产批内，各次送入烘干房的产品子批，因产线工人操作不规范，在烘干房内的 **位置不同**、烘干房 **频繁开启关闭** 等不可控因素，导致 **相同产品、相同生产批** 的烘干效果不一致，出现过湿或过干等次品。

**信息与决策延时（系统缺失）：** 整个 QC 环节 **基本无系统接入**，全靠人工检测、统计、录入。典型情况：**中午完成烘干** 的批次，**傍晚** 上层管理才收到检验结果；此时烘干房已开始下一轮烘干、产线已开始新批生产， **不合格品难以及时返烘或分流，只能现场堆积**。

#### 1.4.2 问题三层分解


| 层级                | 内容                           | Phase 1 系统能做什么                        | 需现场管理 / 工艺配合的部分                  |
| ----------------- | ---------------------------- | ------------------------------------- | -------------------------------- |
| **第 1 层：工艺/现场变异** | 烘干位置不统一、开关频繁 → 同批含水/水活不一致、次品 | 记录 **烘干子批 + 位置/炉次**，支持缺陷与位置关联分析       | SOP、货位规范、烘干规程（Phase 2 可与 MES 深化） |
| **第 2 层：检验与记录**   | 仪器不联网 → 纸笔；纸质标签 → 无实时系统状态    | **平板录入读数**、自动判定、电子 Hold、消灭 Excel 二次汇总 | 仪器自动采集列入 Phase 2                 |
| **第 3 层：决策与处置闭环** | 汇总 + 上报慢 → 错过返烘时间窗 → 堆积      | **提交即通知**、处置工作流、待检/Hold 看板            | 返烘排期、产能协调需客户流程配合                 |


客户强调的 **「QC 延时」** 主要指 **第 2 + 第 3 层**；**次品多发** 则与 **第 1 层** 强相关。产品叙事建议：

> 先消灭「半天才知道、无法及时处置」；再用数据支撑「哪个烘干位置缺陷率高」的工艺改进。

#### 1.4.3 延时发生在哪些环节（时间线）


| 顺序  | 环节            | 现状               | 对延时的贡献                        |
| --- | ------------- | ---------------- | ----------------------------- |
| 1   | 产线完成 → 分批入烘干房 | 子批多、位置随意、开关不可控   | 不直接拖慢「报告」，但 **子批多 → 检验工作量放大** |
| 2   | 烘干完成          | 无系统事件「某子批已出房、待检」 | 管理层不知 **待检队列深度**              |
| 3   | QC 逐批仪器检测     | 人工测量、纸笔          | 检验耗时 + 抄写风险                   |
| 4   | 贴合格/不合格纸标     | 现场可见，系统不可见       | ERP/管理层仍不可见                   |
| 5   | 管理员汇总 Excel   | 二次整理             | **主要信息延时（数小时）**               |
| 6   | 提交上层决策        | 邮件/微信/开会         | 再等一轮                          |
| 7   | 返烘/粉碎等处置      | 决策时烘干房、产线已占用     | **处置失效 → 物理堆积**               |


「中午烘干完、傍晚才收到结果」往往 **不是仪器检测需要半天**，而是 **检验 → 汇总 → 上报 → 决策** 链路上缺少 **单批实时质量状态**，全部挤在 **Excel 汇总** 这一瓶颈。

#### 1.4.4 不合格品堆积的因果链

```text
检验完成 ──(无实时上报)──> 数小时后才到管理层
                              │
                              ▼
                    烘干房已下一炉 / 产线已新批
                              │
                              ▼
              返烘或分流错过时间窗 → 不合格品只能堆积
```

叠加因素：

- 在管理层决定前，子批 **无系统 Hold**，存在误流转或混放风险；
- 无 **烘干炉档期/待检队列** 视图，决策时不知返烘是否可插入。

#### 1.4.5 业务对象：生产批与烘干子批


| 概念   | 建议命名          | 说明                                       |
| ---- | ------------- | ---------------------------------------- |
| 生产批  | `生产批 / Lot`   | 产线一次投产对应的主批号                             |
| 烘干子批 | `烘干子批 / 干燥子批` | 每次从产线送入烘干房的一车/一盘/一框；**检验与 Hold 建议挂在此层级** |
| 检验单  | `检验单`         | 通常对应一个烘干子批的一次检验                          |
| 质量状态 | 状态机字段         | 如：待检 / 检验中 / 合格 / 不合格-Hold / 处置中 / 已关闭   |


**调研必确认：** 现场纸质标签贴的是 **生产批号** 还是 **烘干子批号**？若仅有生产批号，同批内「哪一托过干」难以区分。

#### 1.4.6 目标业务流程与状态机（Phase 1 设计参考）

```text
产线完工 → 创建/扫描 生产批
  → 每批进烘干房：登记 烘干子批 +（可选）位置/炉号/层架 + 进房时间
  → 出烘干房 → 状态 = 待检（进入 QC 队列，可触发逾期提醒）
  → QC 员：选子批 → 录入仪器读数 → 提交
        ├─ 合格 → 允许进入下道工序（与现 ERP 规则对齐）
        └─ 不合格 → 自动 Hold + 通知质量/生产负责人
              → 在线选择处置：返烘 / 粉碎回线 / 报废 / 让步 等
              → 记录决策人、时间、原因 → 关闭子批
```

**需客户确认的业务规则：**

- 不合格子批 Hold 期间是否 **禁止移库或进入下道工序**？
- 返烘是否 **沿用原子批号** 还是生成新子批号？
- 粉碎回线是否记为 **新生产批** 并关联原批？

#### 1.4.7 针对本场景的系统能力优先级


| 优先级         | 能力                | 说明                             |
| ----------- | ----------------- | ------------------------------ |
| **P0 必做**   | 烘干子批登记 + 扫码/选批    | 进房、出房关键节点可登记                   |
| **P0 必做**   | 检验模板 + 人工录入仪器读数   | 大数字键盘、上下限自动判定；可附仪器/标签照片        |
| **P0 必做**   | 提交即更新状态 + 通知      | 不合格自动 Hold；管理层 **分钟级** 可见（非傍晚） |
| **P0 必做**   | 消灭管理员 Excel 汇总    | 看板实时汇总，管理员角色转为异常处理             |
| **P0 必做**   | 简化处置工作流           | 返烘 / 粉碎 / 报废 / 让步等选项可配置        |
| **P0 必做**   | 待检队列看板            | 已出房待检数、最长等待时间、逾期未检高亮           |
| **P1 建议**   | 记录烘干位置/炉次         | 下拉即可；支撑后续缺陷与位置关联分析             |
| **P1 建议**   | 烘干炉简单档期标记         | 当前炉次、预计结束，辅助返烘决策               |
| **Phase 2** | 仪器自动采集、温湿度曲线、自动排程 | 避免 Phase 1 scope 膨胀            |


#### 1.4.8 对内对齐（一句话）

> **客户要的不只是电子检验单，而是「烘干子批一检完，管理层立刻能处置，且不再被 Excel 汇总拖半天」；次品根因在现场烘干变异，系统先用子批 + 位置 + 即时 Hold/看板缩短信息与决策时间，再用数据支撑工艺改进。**

## 2. 产品定位与边界

### 2.1 我们做什么

**「QC + 生产现场数据采集平台」**（可演进为 MES-lite），首要目标：

- 现场 **实时录入** 检验结果与关键工艺参数
- **批次（Lot/Batch）** 全链路可追溯
- **Hold / Release** 等处置流程在线闭环
- 管理端 **实时看板**（合格率、Hold 数量、缺陷趋势、**待检烘干子批队列** 等）
- **优先覆盖** [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议) 烘干后检验与处置闭环

### 2.2 我们暂不做什么（Phase 1）

- 不替换现有 ERP 的总账、应收应付、税务
- 不做完整 MRP、高级排程（留给后续阶段）
- 不追求「大而全 ERP」第一版

### 2.3 与现有 ERP 的关系（Overlay 策略）

```
┌─────────────────────────────────────┐
│  新系统：制造 / QC / 追溯（真相源）   │
│  · 现场检验、Hold/Release、批次事件   │
└──────────────┬──────────────────────┘
               │ 同步（API / 文件 / 人工确认）
               ▼
┌─────────────────────────────────────┐
│  现有 ERP：订单、库存账务、采购、财务  │
└─────────────────────────────────────┘
```

**开工前必须明确：**

- 物料、客户、供应商等 **主数据以谁为准**
- 检验合格后 **库存/可发货状态由谁更新**
- Hold 是否必须 **回写或锁定** 现 ERP 中的库存

## 4. 如何起步（Phase 0 行动清单）

在写业务代码之前，按顺序完成以下事项。每一项未完成，都会在设计或开发阶段返工。

### 4.1 现场调研（必做）


| 活动                    | 时长建议    | 产出                                  |
| --------------------- | ------- | ----------------------------------- |
| 跟生产线至少 1 个完整班次        | 1 天     | 过程 QC 触点时间、记录方式、谁签字                 |
| 跟 QC 员（来料 + 成品）       | 1 天     | 检验项清单、抽样规则、纸质表单拍照                   |
| **跟烘干房 + 烘干后 QC**（优先） | 1 天     | 子批划分、位置命名、出房待检流程、仪器型号与耗时、Excel 汇总方式 |
| 跟仓库（收料、发货）            | 0.5–1 天 | 批次号如何生成、贴标、与 ERP 录入关系               |
| 访谈生产经理 / 质量负责人        | 2–4 小时  | 痛点排序、审批链、合规要求                       |
| 访谈 IT 或 ERP 管理员       | 2 小时    | 现 ERP 品牌版本、接口、主数据导出方式               |


**原则：** 不仅听管理层，必须听 **QC 技术员、线长、仓库文员**——他们才是日常用户。

### 4.2 交付物（Phase 0 结束时应具备）

- **端到端流程图**（从原料入库 → 来料检 → 投产 → **烘干子批** → **烘干后 QC** → 包装 → 成品检 → 入库 → 发货）
- **烘干后 QC 专项流程图**（与 [1.4.6](#146-目标业务流程与状态机phase-1-设计参考) 对照，标注角色与时长）
- **QC 节点清单**（每个节点：触发条件、检验项、频次、责任人、现行表单）
- **批次号规则说明**（编码规则、在哪些环节打印/扫描）
- **角色与权限矩阵**（谁可录入、谁可 Hold、谁可 Release）
- **与现 ERP 集成说明**（同步字段、频率、冲突处理）
- **MVP 范围文档 + 验收 KPI**（客户签字或邮件确认）
- **主数据初版清单**（物料、供应商、产线、检验模板）

### 4.3 内部准备

- 确定技术栈与部署方式（云 / 本地、车间网络、是否需离线）
- 确定第一版终端（iPad / Android 平板 / 手机 PWA / 扫码枪）
- 确定项目对接人（客户方质量 + IT 各一人）

| 编号   | 问题                                                       | 填写 / 备注                                                      |
| ---- | -------------------------------------------------------- | ------------------------------------------------------------ |
| B-01 | 客户最想解决的 Top 3 痛点是什么？（延迟录入 / 追溯慢 / Hold 不及时 / 烘干后堆积 / 其他） | 已知：**烘干后 QC 延时 + 不合格品堆积**（见 [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议)） |

| B-05 | 预计同时在线使用的 QC/现场用户数？                                      | 20                                                           |

### 5.2 生产与 QC 流程（流程映射）

为 **每一条产品线或车间**，描述完整路径，并标注 QC 节点。


| 编号   | 问题                                        | 填写 / 备注                                                     |
| ---- | ----------------------------------------- | ----------------------------------------------------------- |
| F-01 | 有哪些产品线 / 车间 / 产线？各线工艺是否差异大？               | 有两条以上的产品线，各个产品工艺差异较大，所以每一次更换产品时都需要清洗产品线上的机器                 |
| F-02 | **来料检验（IQC）**：哪些物料必须检？哪些可免检？              |                                                             |
| F-03 | IQC 触发时机：到货即检 / 入库前 / 其他？                 | 产品完成烘干后进行QC                                                 |
| F-04 | **过程检验（IPQC）**：有哪些工序节点（烘干、混合、金属检测等）？      | 烘干                                                          |
| F-05 | IPQC 频次：每批首件 / 每 30 分钟 / 每班 / 其他？         | 每批首件                                                        |
| F-06 | **成品检验（FQC）**：在包装前、包装后还是入库前？              | 包装前                                                         |
| F-07 | 是否存在 **驻厂检 / 第三方检 / 客户批检**？               | 不存在                                                         |

## 6. Phase 1 MVP 功能范围建议

第一版只做「能替代纸、能闭环、能追溯」，克制范围。**首期以 [1.4 烘干后 QC](#14-客户现场痛点烘干后-qc-延时第一次会议) 为主战场**，其它 QC 节点可复用同一套模板与子批模型后续上线。

### 6.1 建议包含


| 模块   | 功能要点                                                |
| ---- | --------------------------------------------------- |
| 主数据  | 物料（同步或导入）、产线、**烘干房/炉位**、检验模板（含烘干后水活/水分等项）           |
| 批次管理 | **生产批** + **烘干子批**；进房/出房登记；可选位置/炉号；与纸质标签号对齐         |
| 检验执行 | 待检队列 → 选子批 → **人工录入仪器读数** → 自动判定 → 提交；支持照片          |
| 处置   | 不合格 **自动 Hold** + 通知；返烘 / 粉碎回线 / 报废 / 让步（可配置）；审批简化版 |
| 追溯   | 生产批 ↔ 烘干子批 ↔ 检验记录 ↔ 处置记录                            |
| 看板   | **待检子批数/最长等待**、Hold 列表、合格率、逾期未检提醒；**替代 Excel 例行汇总** |
| 审计   | 操作日志、关键操作电子签名（若客户有要求）                               |
| 集成   | 至少一种：CSV 导出 或 单接口同步物料/库存状态                          |


### 6.2 建议不包含（避免 scope creep）

- 完整工单排程、设备 OEE、复杂 MRP
- 总账、应收应付、税务
- 与所有检测仪器全自动对接（可列 Phase 2）
- 多工厂、多语言（除非合同明确要求）

## 8. 技术架构原则（概要）

### 8.1 核心设计原则

1. **事件/事务导向**：不仅存最终状态，保留检验提交、Hold、Release 等事件，便于审计与追溯。
2. **批次为中心**：检验单、照片、处置记录挂载 **烘干子批 ID**；生产批用于聚合与追溯。
3. **Mobile / Tablet First**：现场能用是第一优先级；考虑 PWA + 弱网离线。
4. **模板驱动检验**：检验项配置化，减少为每个 SKU 写死页面。
5. **审计默认开启**：谁、何时、做了什么、改前改后（食品行业刚需）。

### 8.2 技术栈（可按团队调整）


| 层级  | 常见选型                  |
| --- | --------------------- |
| 前端  | React + Vite，平板适配，PWA |
| 后端  | FastAPI / Node.js 等   |
| 数据库 | PostgreSQL / MySQL    |
| 部署  | 按客户网络条件选择 Docker 本地或云 |


> 语言栈不如 **数据模型与集成边界** 重要；优先保证批次模型和 ERP 接口契约正确。

### 8.3 核心实体关系（概念）

```
物料 ──< 检验模板
生产批 ──< 烘干子批 ──< 检验单 ──< 检验项结果
烘干子批 ──< 质量事件（Hold / 返烘 / 粉碎 / 报废…）
烘干子批 ──> 烘干房位置、进房/出房时间
生产批 ──> 原料批关联（追溯链，后期扩展）
生产批 / 子批 ──?── 现 ERP 工单 / 库存 ID
```

与本客户相关的主路径见 [1.4.5](#145-业务对象生产批与烘干子批)、[1.4.6](#146-目标业务流程与状态机phase-1-设计参考)。

3. 检验标准从哪里查？有没有经常变更？
4. 不合格时你下一步做什么？找谁签字？
5. 如果给你一台平板，你最希望少填哪些字？
6. 车间哪里信号最差？

### 9.2 线长 / 生产主管

1. 开工前必须完成哪些 QC？没完成能否开工？
2. 过程检多久做一次？谁提醒？
3. 产线换型、换 SKU 时批次号怎么处理？
4. 你最常因为 QC 数据不准确遇到什么问题？

### 9.3 仓库

1. 到货时批号怎么贴？和 ERP 入库单如何对应？
2. Hold 的货放在哪里？系统里现在能否看出 Hold？
3. 发货前是否核对成品检状态？

### 9.4 质量经理

1. 当前最大质量风险是什么？
2. 审厂或客诉时，追溯要准备哪些材料？准备多久？
3. 哪些人必须有 Release 权限？
4. Phase 1 三个月内，你最想看到哪三个数字？

### 9.5 ERP / IT

1. ERP 品牌版本、模块清单、是否有 API？
2. 能否提供物料、库存、工单样例导出？
3. 未来是否接受「QC 系统驱动库存状态」？

### 9.6 烘干房 / 烘干后 QC（第一次会议后的专项追问）

1. 烘干房有几间/几炉？位置、炉次如何称呼？
2. 从产线到烘干房，一般多少车/批？子批如何编号？标签贴在哪？
3. 出烘干房后，多久内必须完成检验？现在实际平均多久？
4. 仪器测什么（水活 Aw、水分 % 等）？每台每天大约测多少子批？
5. 纸笔记录表能否提供照片？合格/不合格如何判定？
6. 谁把数据抄进 Excel？每天几次？上层管理通过什么渠道收表？
7. 返烘、粉碎回线、报废各占比？谁批准？返烘一般要在出房后几小时内？
8. 不合格货现在堆在哪？Hold 期间会不会被误领用？
9. 合格子批下一步是包装还是暂存？与 ERP 哪一步对应？
10. 若系统取消 Excel 汇总，该管理员是否愿意改为只看板 + 处理异常？
