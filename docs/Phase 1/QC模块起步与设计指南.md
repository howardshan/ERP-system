## 1. 背景与问题定义

### 1.1 客户现状

- 已在使用一套 ERP，在 **财务、采购、销售** 等方面基本可用。
- **生产制造、车间、QC、批次追溯** 能力薄弱，满意度低。
- QC 环节以 **纸质记录为主**，事后由管理人员 **手工录入 ERP** 或汇总 Excel，导致：
  - 数据滞后（T+0 现场 → 数小时后甚至更晚才到管理层，详见 [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议））
  - 库存/放行状态与现场真实质量不一致
  - 追溯、客诉、召回响应慢
  - 管理看板无法反映「当前产线真实状态」
  - 不合格品因处置决策滞后而 **现场堆积**（烘干房、产线已投入后续任务）

### 1.2 行业特征（宠物零食 / 食品级制造）


| 特征    | 对系统的要求               |
| ----- | -------------------- |
| 强批次管理 | 原料批、生产批、成品批全链路关联     |
| 强 QC  | 来料 / 过程 / 成品多节点检验    |
| 强追溯   | 由成品批反查原料批、产线、人员、时段   |
| 强合规   | 检验记录留存、审计追踪、不合格品处置留痕 |
| 现场环境  | 手套、油污、少打字、多扫码/点选     |


### 1.3 核心问题（一句话）

> **生产现场质量数据没有实时数字化**，而不是「缺少一套新财务系统」。

### 1.4 客户现场痛点：烘干后 QC 延时（第一次会议）

本节记录与客户管理人员第一次会议中确认的 **首要痛点场景**。Phase 1 MVP 应优先覆盖此链路，其它 QC 节点（来料检、成品包装检等）可在同架构上扩展。

#### 1.4.1 现场流程描述（现状）

```text
产线生产完成
  → 同一生产批的产品分批送入烘干房
  → 烘干完成（各子批出房时间可能不同）
  → QC 员用仪器逐批检测（仪器无法接入系统）
  → 人工读数 + 纸笔记录
  → 在对应批次上贴「合格 / 不合格」纸质标签
  → 管理人员将数据与结果汇总至 Excel
  → 提交上层管理决策
  → 处置：返烘 / 粉碎后重新投入产线 / 其它方式
```

**工艺与质量变异（次品来源）：** 同一生产批内，各次送入烘干房的产品子批，因产线工人操作不规范，在烘干房内的 **位置不同**、烘干房 **频繁开启关闭** 等不可控因素，导致 **相同产品、相同生产批** 的烘干效果不一致，出现过湿或过干等次品。

**信息与决策延时（系统缺失）：** 整个 QC 环节 **基本无系统接入**，全靠人工检测、统计、录入。典型情况：**中午完成烘干** 的批次，**傍晚** 上层管理才收到检验结果；此时烘干房已开始下一轮烘干、产线已开始新批生产， **不合格品难以及时返烘或分流，只能现场堆积**。

#### 1.4.2 问题三层分解


| 层级                | 内容                           | Phase 1 系统能做什么                        | 需现场管理 / 工艺配合的部分                  |
| ----------------- | ---------------------------- | ------------------------------------- | -------------------------------- |
| **第 1 层：工艺/现场变异** | 烘干位置不统一、开关频繁 → 同批含水/水活不一致、次品 | 记录 **烘干子批 + 位置/炉次**，支持缺陷与位置关联分析       | SOP、货位规范、烘干规程（Phase 2 可与 MES 深化） |
| **第 2 层：检验与记录**   | 仪器不联网 → 纸笔；纸质标签 → 无实时系统状态    | **平板录入读数**、自动判定、电子 Hold、消灭 Excel 二次汇总 | 仪器自动采集列入 Phase 2                 |
| **第 3 层：决策与处置闭环** | 汇总 + 上报慢 → 错过返烘时间窗 → 堆积      | **提交即通知**、处置工作流、待检/Hold 看板            | 返烘排期、产能协调需客户流程配合                 |


客户强调的 **「QC 延时」** 主要指 **第 2 + 第 3 层**；**次品多发** 则与 **第 1 层** 强相关。产品叙事建议：

> 先消灭「半天才知道、无法及时处置」；再用数据支撑「哪个烘干位置缺陷率高」的工艺改进。

#### 1.4.3 延时发生在哪些环节（时间线）


| 顺序  | 环节            | 现状               | 对延时的贡献                        |
| --- | ------------- | ---------------- | ----------------------------- |
| 1   | 产线完成 → 分批入烘干房 | 子批多、位置随意、开关不可控   | 不直接拖慢「报告」，但 **子批多 → 检验工作量放大** |
| 2   | 烘干完成          | 无系统事件「某子批已出房、待检」 | 管理层不知 **待检队列深度**              |
| 3   | QC 逐批仪器检测     | 人工测量、纸笔          | 检验耗时 + 抄写风险                   |
| 4   | 贴合格/不合格纸标     | 现场可见，系统不可见       | ERP/管理层仍不可见                   |
| 5   | 管理员汇总 Excel   | 二次整理             | **主要信息延时（数小时）**               |
| 6   | 提交上层决策        | 邮件/微信/开会         | 再等一轮                          |
| 7   | 返烘/粉碎等处置      | 决策时烘干房、产线已占用     | **处置失效 → 物理堆积**               |


「中午烘干完、傍晚才收到结果」往往 **不是仪器检测需要半天**，而是 **检验 → 汇总 → 上报 → 决策** 链路上缺少 **单批实时质量状态**，全部挤在 **Excel 汇总** 这一瓶颈。

#### 1.4.4 不合格品堆积的因果链

```text
检验完成 ──(无实时上报)──> 数小时后才到管理层
                              │
                              ▼
                    烘干房已下一炉 / 产线已新批
                              │
                              ▼
              返烘或分流错过时间窗 → 不合格品只能堆积
```

叠加因素：

- 在管理层决定前，子批 **无系统 Hold**，存在误流转或混放风险；
- 无 **烘干炉档期/待检队列** 视图，决策时不知返烘是否可插入。

#### 1.4.5 业务对象：生产批与烘干子批


| 概念   | 建议命名          | 说明                                       |
| ---- | ------------- | ---------------------------------------- |
| 生产批  | `生产批 / Lot`   | 产线一次投产对应的主批号                             |
| 烘干子批 | `烘干子批 / 干燥子批` | 每次从产线送入烘干房的一车/一盘/一框；**检验与 Hold 建议挂在此层级** |
| 检验单  | `检验单`         | 通常对应一个烘干子批的一次检验                          |
| 抽样组  | `抽样组 / Sampling Group` | 按 SKU 规则（如每 N 车）划分的子批集合；**系统随机选车检验，结果可级联至同组**（见 [1.5.4](#154-抽样组与级联-reject)） |
| 质量状态 | 状态机字段         | 如：等待烘干 / 正在烘干 / 待检 / 检验中 / 合格 / 不合格-Hold / 室温风干 / 处置中 / 已关闭   |


**调研必确认：** 现场纸质标签贴的是 **生产批号** 还是 **烘干子批号**？若仅有生产批号，同批内「哪一托过干」难以区分。

#### 1.4.6 目标业务流程与状态机（Phase 1 设计参考）

```text
产线完工 → 创建/扫描 生产批 + 登记 N 个烘干子批
  → 子批进烘干房：登记位置/炉号/层架 + 进房时间 + SKU 标准烘干时长（倒计时/逾期提醒）
  → 出烘干房 → 状态 = 待检（进入 QC 队列）
  → 系统按 SKU 抽样规则（如每 3 车 1 检）形成抽样组，随机指定「待取样子批」并提示 QC 员
  → QC 员：到指定子批取样 → 录入仪器读数 → 自动判定（支持双限 / 仅上限等）→ 提交
        ├─ 合格 → 同组关联子批按规则同步 Pass（待客户确认）→ 放行 → 等待包装
        └─ 不合格 → 同组关联子批按规则级联 Reject/Hold（待客户确认）
              → 自动 Hold + 通知质量/生产负责人
              → 在线选择处置：返烘 / 室温风干 / 粉碎回线 / 报废 / 让步 等
              → 记录决策人、时间、原因 → 关闭子批
```

> **v1.2 更新：** 抽样选车、级联 reject、烘干时长控制、任务 Dashboard 等细节见 [1.5](#15-第二次会议对齐要点)。

**需客户确认的业务规则：**

- 不合格子批 Hold 期间是否 **禁止移库或进入下道工序**？
- 返烘是否 **沿用原子批号** 还是生成新子批号？
- 粉碎回线是否记为 **新生产批** 并关联原批？

#### 1.4.7 针对本场景的系统能力优先级


| 优先级         | 能力                | 说明                             |
| ----------- | ----------------- | ------------------------------ |
| **P0 必做**   | 烘干子批登记 + 扫码/选批    | 进房、出房关键节点可登记                   |
| **P0 必做**   | 检验模板 + 人工录入仪器读数   | 大数字键盘、**可配置判定模式**（双限 / 仅上限等）；可附仪器/标签照片 |
| **P0 必做**   | **SKU/子批烘干时长控制**   | 标准时长、预计出房、剩余时间、逾期高亮（见 [1.5.1](#151-烘干时长控制)） |
| **P0 必做**   | **任务型 Dashboard**   | 六类队列：等待烘干 / 正在烘干 / 待检 / 检验通过 / 检验未通过 / 室温风干（见 [1.5.3](#153-任务-dashboard)） |
| **P0 必做**   | 提交即更新状态 + 通知      | 不合格自动 Hold；管理层 **分钟级** 可见（非傍晚） |
| **P0 必做**   | 消灭管理员 Excel 汇总    | 看板实时汇总，管理员角色转为异常处理             |
| **P0 必做**   | 简化处置工作流           | 返烘 / 室温风干 / 粉碎 / 报废 / 让步等选项可配置 |
| **P0 必做**   | 待检队列看板            | 已出房待检数、最长等待时间、逾期未检高亮           |
| **P0 必做**   | **系统随机抽样 + 选车指引** | 按 SKU 配置「每 N 车一检」；系统随机指定取样子批（见 [1.5.2](#152-随机抽样与选车指引)） |
| **P0/P1**   | **抽样组级联 Reject**   | 抽中样 fail → 关联子批全部 reject（规则待客户确认，见 [1.5.4](#154-抽样组与级联-reject)） |
| **P1 建议**   | 记录烘干位置/炉次         | 下拉即可；支撑后续缺陷与位置关联分析             |
| **P1 建议**   | 烘干炉简单档期标记         | 当前炉次、预计结束，辅助返烘决策               |
| **P1 建议**   | **等待打包 WIP 计数**   | 检验通过、待包装的生产批/子批数量（见 [1.5.5](#155-等待打包与-sku-统计)） |
| **Phase 1.5** | **SKU 维度统计**       | 平均实际烘干时长、通过率等（见 [1.5.5](#155-等待打包与-sku-统计)） |
| **Phase 2** | 仪器自动采集、QC 3 天预测、自动排程 | 预测为启发式分析，非 MRP 排程（见 [1.5.6](#156-qc-工作量预测)） |


#### 1.4.8 对内对齐（一句话）

> **客户要的不只是电子检验单，而是「烘干子批一检完，管理层立刻能处置，且不再被 Excel 汇总拖半天」；次品根因在现场烘干变异，系统先用子批 + 位置 + 即时 Hold/看板缩短信息与决策时间，再用数据支撑工艺改进。**

### 1.5 第二次会议对齐要点

本节记录与客户 **第二次会议** 对齐/补充的需求。在 [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议)「消灭 QC 延时」基础上，客户进一步明确了 **过程控制（烘干时长）**、**抽样规则**、**任务看板** 与 **分析预测** 等能力。以下 8 条为会议输入；其中带 ⚠️ 的项需在开发前与客户 **书面确认**。

#### 1.5.1 烘干时长控制

**会议需求：** 各批次 / SKU 进入烘干房后，需有明确的烘干时间控制。

**设计含义：**

- 每个 SKU 维护 **标准烘干时长**（SOP 参考值，分钟）。
- 每个烘干子批进房时继承 SKU 标准值，必要时允许 **单次覆盖**（如返烘后重新设定 `expected_dry_minutes`）。
- 系统展示：**预计出房时间**、**剩余烘干时间**、**逾期**（超过标准时长仍未出房）。
- 与 [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议) 第 1 层（位置/开关导致变异）结合：按「SKU + 位置」统计可对比「标准 vs 实际」烘干时长。

**待客户确认：**

- 逾期是否 **禁止出房**，还是仅提醒？
- 同一 SKU 不同生产批是否允许调整标准时长？

#### 1.5.2 随机抽样与选车指引

**会议需求：** 并非每车都检——例如 **每三车一取**，且 **不同 SKU 规则可不同**；随机取样时由 **系统生成随机数并告知员工去哪车取样**。

**设计含义：**

- 检验模板 / SKU 主数据增加 **抽样间隔**（如 `sample_every_n_sub_lots = 3`）。
- 生产批下子批按进房或创建顺序划入 **抽样组**（Sampling Group）；每组 N 车中系统 **随机指定 1 车** 为「待取样 / 待检验」。
- QC 员界面突出显示：**「请到第 X 车（子批号 XXX）取样」**——减少人工决定带来的偏差与争议。
- 未被抽中的子批进入 **等待组结果** 状态，不重复检验（若组级 Pass 传播规则确认，见 [1.5.4](#154-抽样组与级联-reject)）。

**待客户确认 ⚠️：**

- 「每三车」按 **生产批内子批顺序** 还是 **全局连续三车** 划分？
- 随机是否需在组形成时 **立即固定**（可审计），还是每次打开待检队列时重新抽？
- 抽中车 Pass 后，同组其余车是否 **自动 Pass**？

#### 1.5.3 任务 Dashboard

**会议需求：** 统一的任务 Dashboard，至少包含以下六类视图（含计数与列表钻取）：

| 客户表述 | 建议系统状态 / 队列 | 说明 |
| -------- | ------------------- | ---- |
| **等待烘干** | `created`（已登记、未进烘干房） | 产线已产出子批，等待放入烘干房 |
| **正在烘干** | `drying` | 在烘干房内，显示剩余/预计出房时间 |
| **待检** | `pending` / `inspecting` | 已出房；可拆「待取样 / 待录 Aw」子状态 |
| **检验通过** | `passed`（待 release） | 已通过检验，等待放行 / 进入包装 |
| **检验未通过** | `hold` / `disposing` | Hold 或处置中 |
| **in-door 风干**（室温风干） | `room_temp_drying` | 不合格处置路径之一；计时结束后回待检 |

**设计含义：** 替代 Excel 汇总的核心界面；管理层与现场主管 **一屏掌握 WIP**。与 [5.11 D-07～D-12](#511-报表与看板) 对应。

#### 1.5.4 抽样组与级联 Reject

**会议需求：** Reject 场景下，**reject 样品所关联的批次需全部 reject**。

**设计含义：**

- 引入 **抽样组** 实体：同一组内子批共享一次检验结论（或级联结论）。
- 当 **被抽中子批** 检验 Fail 时，触发 **组级联 Reject**：同组所有子批自动 Hold / Reject，无需逐车再检。
- 需写入 **质量事件** 留痕：级联来源子批、操作人、时间、规则版本。

**待客户确认 ⚠️（开发阻塞项）：**

- 「关联批次」= **同抽样组全部子批**，还是 **整个生产批**？
- 级联 Reject 是否包含 **已自动 Pass 的同组子批**（追溯性改判）？
- 级联后处置是 **整组统一处置** 还是 **逐车单独处置**？

#### 1.5.5 等待打包与 SKU 统计

**会议需求（6）：** 展示 **等待打包** 的生产批 / 生产子批数量。

**会议需求（7）：** **特定 SKU** 的平均烘干时间、通过率等数据统计。

**设计含义：**

- **等待打包：** 统计处于 `passed`（或 `released`，待客户定义）且下一工序为包装分拣的子批/生产批数量；Dashboard 独立卡片。
- **SKU 统计（Phase 1.5）：** 按 SKU + 时间窗汇总——平均 **实际烘干时长** vs 标准时长、检验 **通过率 / 不合格率**、可选按烘干位置交叉分析。支撑工艺改进叙事（[1.4.2](#142-问题三层分解) 第 1 层）。

**待客户确认：**

- 「等待打包」统计单位：**生产批** 还是 **烘干子批**？
- 起点状态：`passed` 还是 `released` / `ready_for_pack`？

#### 1.5.6 QC 工作量预测

**会议需求（8）：** **QC 数量 3 天预测**。

**设计含义：**

- **非** 完整 MRP / 自动排程；而是基于 **在途子批**（正在烘干 + 已知计划投产）+ **历史日均待检量** 的 **启发式预测**。
- 输出示例：未来 3 天预计 **待检子批数**、**预计检验单数**、可选 **所需 QC 工时**（按每单耗时假设）。
- 建议列入 **Phase 2**；Phase 1 先积累数据与 Dashboard 六队列。

**待客户确认：**

- 预测主要用于 **QC 排班**、**管理汇报** 还是 **物料/包装产能协调**？
- 是否需要接入 **生产计划 / 工单** 作为输入？

#### 1.5.7 检验判定：双限与仅上限

**会议需求（2）：** 部分检验项有 **硬性双限**（上下限）；部分 **只有上限、无下限**（upper line only）。

**设计含义：**

- 检验模板每项增加 **判定模式**：`both_limits`（闭区间）/ `upper_only`（仅 `value > upper` 为 fail）/ `lower_only` / `informational`（仅记录、不自动判 fail）。
- 自动判定逻辑须支持 **可空 `lower_limit`**；不可再假设所有项均为双限。

#### 1.5.8 能力优先级（第二次会议更新）

| 优先级 | 能力 | 对应会议条目 |
| ------ | ---- | ------------ |
| **P0** | 烘干时长控制 + 逾期可见 | §1.5.1 |

## 2. 产品定位与边界

### 2.1 我们做什么

**「QC + 生产现场数据采集平台」**（可演进为 MES-lite），首要目标：

- 现场 **实时录入** 检验结果与关键工艺参数
- **批次（Lot/Batch）** 全链路可追溯
- **Hold / Release** 等处置流程在线闭环
- 管理端 **任务型 Dashboard**（等待烘干 / 正在烘干 / 待检 / 检验通过 / 检验未通过 / 室温风干等，见 [1.5.3](#153-任务-dashboard)）
- **SKU 烘干时长控制**、**可配置检验判定**、**系统随机抽样选车**
- **优先覆盖** [1.4](#14-客户现场痛点烘干后-qc-延时第一次会议) 烘干后检验与处置闭环；[1.5](#15-第二次会议对齐要点) 补充过程控制与分析能力

### 2.2 我们暂不做什么（Phase 1）

- 不替换现有 ERP 的总账、应收应付、税务
- 不做完整 MRP、高级排程（留给后续阶段）
- 不追求「大而全 ERP」第一版

### 2.3 与现有 ERP 的关系（Overlay 策略）

```
┌─────────────────────────────────────┐
│  新系统：制造 / QC / 追溯（真相源）   │
│  · 现场检验、Hold/Release、批次事件   │
└──────────────┬──────────────────────┘
               │ 同步（API / 文件 / 人工确认）
               ▼
┌─────────────────────────────────────┐
│  现有 ERP：订单、库存账务、采购、财务  │
└─────────────────────────────────────┘
```

**开工前必须明确：**

- 物料、客户、供应商等 **主数据以谁为准**
- 检验合格后 **库存/可发货状态由谁更新**
- Hold 是否必须 **回写或锁定** 现 ERP 中的库存

## 8. 技术架构原则（概要）

### 8.1 核心设计原则

1. **事件/事务导向**：不仅存最终状态，保留检验提交、Hold、Release 等事件，便于审计与追溯。
2. **批次为中心**：检验单、照片、处置记录挂载 **烘干子批 ID**；生产批用于聚合与追溯。
3. **Mobile / Tablet First**：现场能用是第一优先级；考虑 PWA + 弱网离线。
4. **模板驱动检验**：检验项配置化（含 **判定模式**、可空下限），减少为每个 SKU 写死页面。
5. **抽样组与结论传播**：随机选车、组级 Pass/Fail 传播规则可配置、可审计（见 [1.5.2](#152-随机抽样与选车指引)、[1.5.4](#154-抽样组与级联-reject)）。
6. **审计默认开启**：谁、何时、做了什么、改前改后（食品行业刚需）。

### 8.2 技术栈（可按团队调整）


| 层级  | 常见选型                  |
| --- | --------------------- |
| 前端  | React + Vite，平板适配，PWA |
| 后端  | FastAPI / Node.js 等   |
| 数据库 | PostgreSQL / MySQL    |
| 部署  | 按客户网络条件选择 Docker 本地或云 |


> 语言栈不如 **数据模型与集成边界** 重要；优先保证批次模型和 ERP 接口契约正确。

### 8.3 核心实体关系（概念）

```
物料(SKU) ──< 检验模板（limit_type, lower?, upper）
物料(SKU) ──> standard_drying_minutes（标准烘干时长）
生产批 ──< 抽样组 ──< 烘干子批（expected_dry_minutes, status）
生产批 ──< 烘干子批 ──< 检验单 ──< 检验项结果
烘干子批 ──< 质量事件（check_in / check_out / inspection_* / cascade_reject / disposition_*）
烘干子批 ──> 烘干房位置、进房/出房时间
生产批 ──> 原料批关联（追溯链，后期扩展）
生产批 / 子批 ──?── 现 ERP 工单 / 库存 ID
```

与本客户相关的主路径见 [1.4.5](#145-业务对象生产批与烘干子批)、[1.4.6](#146-目标业务流程与状态机phase-1-设计参考)、[1.5](#15-第二次会议对齐要点)。

1. 当前最大质量风险是什么？
2. 审厂或客诉时，追溯要准备哪些材料？准备多久？
3. 哪些人必须有 Release 权限？
4. Phase 1 三个月内，你最想看到哪三个数字？

### 9.5 ERP / IT

1. ERP 品牌版本、模块清单、是否有 API？
2. 能否提供物料、库存、工单样例导出？
3. 未来是否接受「QC 系统驱动库存状态」？

### 9.6 烘干房 / 烘干后 QC（第一次会议后的专项追问）

1. 烘干房有几间/几炉？位置、炉次如何称呼？
2. 从产线到烘干房，一般多少车/批？子批如何编号？标签贴在哪？
3. 出烘干房后，多久内必须完成检验？现在实际平均多久？
4. 仪器测什么（水活 Aw、水分 % 等）？每台每天大约测多少子批？
5. 纸笔记录表能否提供照片？合格/不合格如何判定？
6. 谁把数据抄进 Excel？每天几次？上层管理通过什么渠道收表？
7. 返烘、粉碎回线、报废各占比？谁批准？返烘一般要在出房后几小时内？
8. 不合格货现在堆在哪？Hold 期间会不会被误领用？
9. 合格子批下一步是包装还是暂存？与 ERP 哪一步对应？
10. 若系统取消 Excel 汇总，该管理员是否愿意改为只看板 + 处理异常？

### 9.7 第二次会议专项追问

1. 各 SKU 的 **标准烘干时长**（分钟）能否提供清单？返烘后时长如何设定？
2. 哪些检验项 **仅上限、无下限**？请列 SKU × 检验项对照表。
3. 各 SKU 的 **抽样间隔 N**（如每 3 车 1 检）能否提供清单？
4. 「每 N 车」按 **生产批内顺序** 还是 **跨批连续** 计数？
5. 系统随机选车后，QC 员 **能否拒绝重抽**？是否需要主管解锁？
6. 抽中车 **Pass** 后，同组其余车是否 **自动 Pass**？还是需要 spot check？
7. 抽中车 **Fail** 后，**级联 Reject 范围** 是同组全部子批还是整个生产批？
8. 级联 Reject 是否包含 **已经 Pass 的同组子批**？
9. 「**等待烘干**」现场指子批 **已创建未进房**，还是 **已排队等空炉**？请统一叫法。
10. 「**in-door 风干**」与返烘入炉的区别？计时规则与结束条件？
11. 「**等待打包**」从哪个系统状态开始计数？按 **生产批** 还是 **子批**？
12. SKU 统计需要哪些 **默认时间窗**（7 天 / 30 天 / 班次）？
13. **QC 3 天预测** 主要用于排班还是管理汇报？是否需对接生产计划/工单？
14. 任务 Dashboard 六队列是否需要 **大屏电视** 7×24 展示？
