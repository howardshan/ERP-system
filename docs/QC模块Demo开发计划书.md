## 1. 文档目的与定位

### 1.1 为什么要做 Demo

客户在第一次会议中已明确 **烘干后 QC 延时 + 不合格品堆积** 为核心痛点（设计指南 §1.4）。团队在尚未完成现场调研、设计指南 §5 仍有大量「待定」项的前提下，需要通过 **可运行的垂直场景 Demo**：

- 向客户展示 **「检验提交 → 管理层即时可见 → 无需 Excel 汇总」** 的能力；
- 表达合作诚意，并为 **第二次会议 + 现场调研** 争取机会；
- 验证内部对业务流程、批次模型、状态机的理解是否与客户一致。

### 1.2 Demo 不是什么

| Demo | 不是 |
|------|------|
| 可商用的 Phase 1 MVP | 完整 ERP 或 MES |
| 与现 ERP 真实对接的系统 | 通过审厂的合规系统 |
| 检验标准已定稿的生产系统 | 支持全部产品线与全部 QC 节点 |

### 1.3 与设计指南的关系

```text
QC模块起步与设计指南.md  ──►  战略、调研项、Phase 1 MVP 范围
         │
         ▼

         │
         ▼（调研后）
```

## 2. Demo 目标与成功标准

### 2.1 业务目标（对客户）

| 编号 | 目标 | 演示时如何体现 |
|------|------|----------------|
| DG-01 | 让客户认同我们理解 **「中午烘干、傍晚 Excel、堆积」** 痛点 | 开场 2 分钟复述 + 对比时间轴 |
| DG-02 | 展示 **烘干子批** 粒度的实时质量状态 | 待检队列、子批详情 |
| DG-03 | 展示 **人工录入仪器读数 → 自动判定** | QC 平板页大数字录入 |
| DG-04 | 展示 **不合格即时 Hold + 管理端可见** | 不合格路径 + 看板红色 Hold |
| DG-05 | 展示 **处置闭环**（返烘/粉碎/报废/让步） | 经理端处置操作 |
| DG-06 | 引导客户确认 **现场调研** 与 **Phase 1 合作** 意愿 | 结尾议程与待定项清单 |

### 2.2 技术/团队目标（对内）

- 验证 **生产批 → 烘干子批 → 检验单** 数据模型可支撑 UI；
- 产出可复用的前端组件（待检列表、检验录入、看板卡片）作为 Phase 1 骨架；
- 明确调研回来以后 **哪些配置项** 必须从客户处获取（见 §14）。

### 2.3 Demo 成功标准（验收）

- [ ] 可在 **15–20 分钟** 内完整演示「合格路径 + 不合格路径」两条线，无阻塞性 Bug；
- [ ] 在 **平板宽度（≥768px）** 下 QC 录入页可舒适操作（大按钮、少打字）；
- [ ] 检验提交后，管理看板 **无需刷新整页** 即可看到新状态（轮询 ≤5 秒或 WebSocket）；
- [ ] 演示环境可通过 **URL** 在会议现场访问（笔记本投屏 + 可选真实平板）；
- [ ] 界面上 **明显标注**「演示数据 / 配置待定」区域，避免过度承诺。

## 4. 范围定义

### 4.1 In Scope（必须实现）

对齐设计指南 §1.4.7 **P0** 与 §6.1 的烘干后子集：

1. **生产批**：选择或模拟录入批次条码 / 工单条码，创建生产批记录。  
2. **烘干子批**：登记进房时间、出房时间、烘干位置（下拉）、关联生产批。  
3. **待检队列**：出房后子批进入 `待检`，列表展示等待时长。  
4. **烘干后检验**：选子批 → 按 SKU 加载模板 → 录入水活 → 自动判定合格/不合格 → 提交。  
5. **Hold**：不合格自动 Hold，记录时间与操作人。  
6. **管理看板**：待检数、Hold 列表、今日合格/不合格计数、简单合格率。  
7. **处置**：经理对 Hold 子批选择返烘 / 粉碎回线 / 报废 / 让步，填写备注，关闭任务。  
8. **基础追溯**：生产批详情页展示其下所有子批、检验结果、处置记录。  
9. **角色切换**：QC 员界面 vs 管理员界面（简易登录即可）。  
10. **演示种子数据**：一键重置或预置 2 个生产批、若干子批。

### 4.2 Out of Scope（Demo 明确不做）

| 排除项 | 原因 |
|--------|------|
| 来料检验（IQC） | 客户首期痛点在烘干后；F-02 未填 |
| 包装前 FQC 完整流程 | F-06 已知但非本次演示主线 |
| 现 ERP API/CSV 真实对接 | I-* 未调研 |
| 仪器蓝牙/串口采集 | T-06、T-10 |
| 双条码解析规则 | L-04 已知存在，规则未定 |
| 电子签名合规、审计改前改后 | C-*, R-07 未定 |
| 离线 PWA、扫码枪 SDK | U-* 未定 |
| 多产品线并行、洗机换型排程 | F-01 仅作口头扩展 |
| 逾期推送至企业微信 | W-02 |
| 多语言 | B-03 内销 |

### 4.3 P1（时间允许则做，否则口头说明）

- 出房后 **逾期待检** 高亮（如 &gt;2 小时，阈值可配置）。  
- 检验单 **照片上传** UI（可不上传真实文件，仅 UI 占位）。  
- 按烘干 **位置** 筛选看板。  
- 简单 **操作日志** 列表（谁、何时、何种事件）。

## 5. 演示场景与用户故事

### 5.1 角色定义（Demo）

| 角色 | 账号（建议） | 权限 |
|------|-------------|------|
| QC 员 | `qc` | 登记子批、出房待检、录入检验、提交 |
| 质量/生产管理员 | `manager` | 看板、Hold 处置、查看追溯 |
| （可选）烘干房操作工 | 合并到 `qc` 或单独 `dryer` | 仅登记进房/出房 |

### 5.2 用户故事

| ID | 作为 | 我要 | 以便 |
|----|------|------|------|
| US-01 | 烘干房操作 | 扫描/输入批次码并登记子批进房、出房 | 系统知道何时待检 |
| US-02 | QC 员 | 在待检列表中选子批并录入水活 | 代替纸笔记录 |
| US-03 | QC 员 | 提交后立刻看到合格/不合格结果 | 代替纸质合格标 |
| US-04 | 管理员 | 在检验提交后 1 分钟内于看板看到 Hold | 代替 Excel 汇总等待 |
| US-05 | 管理员 | 对 Hold 子批选择返烘或粉碎并记录 | 启动处置闭环 |
| US-06 | 管理员 | 查看某生产批下所有子批与检验历史 | 感受追溯能力 |

### 5.3 演示路径 A：合格（约 3 分钟）

```text
登录 qc → 生产批列表 → 选择「批次 DEMO-20250517-01」
  → 新建烘干子批（位置：烘干房 A-上层，登记出房时间=现在）
  → 待检列表出现该子批
  → 进入检验 → SKU：鸡肉条 → 水活 0.70（限值内）→ 提交
  → 状态变「合格」
切换 manager → 看板：待检 -1，今日合格 +1，无 Hold
```

### 5.4 演示路径 B：不合格 + 处置（约 5 分钟）

```text
登录 qc → 对子批「DEMO-01-D02」检验 → 水活 0.90（超限）→ 提交
  → 自动 Hold，页内提示
切换 manager → 看板 Hold 列表即时出现该子批（强调：无需等 Excel）
  → 进入处置 → 选择「返烘」→ 备注「含水偏高」→ 确认
  → 子批状态「已关闭」
（可选）展示时间轴幻灯片：现状「中午→傍晚」 vs Demo「提交→1 分钟内」
```

## 8. 状态机与业务规则（Demo 版）

### 8.1 烘干子批状态

```text
登记中(可选) → 烘干中(可选) → 待检 → 检验中 → 合格
                              └→ 检验中 → 不合格(Hold) → 处置中 → 已关闭
```

**Demo 简化：** 登记时可直接「出房」进入 `待检`，跳过「烘干中」。

### 8.2 状态迁移表

| 当前状态 | 事件 | 下一状态 | 角色 |
|----------|------|----------|------|
| — | 创建并出房登记 | 待检 | QC |
| 待检 | 开始检验 | 检验中 | QC |
| 检验中 | 提交且合格 | 合格 | QC |
| 检验中 | 提交且不合格 | 不合格(Hold) | 系统 |
| 不合格(Hold) | 开始处置 | 处置中 | Manager |
| 处置中 | 完成处置 | 已关闭 | Manager |

### 8.3 判定规则

- 水活 ∈ [下限, 上限] → 合格；否则不合格。  
- 边界值：Demo 采用 **闭区间**（等于上下限算合格），调研时可改。  
- 提交后 **不允许** QC 修改（Demo）；更正流程列 Phase 1。

### 8.4 业务规则（Demo 假设，调研后可能调整）

| 规则 | Demo 假设 |
|------|-----------|
| 检验挂载层级 | 烘干子批 |
| Hold 后是否 blocking 下道 | 仅 UI 提示，不拦截仓库系统 |
| 返烘是否新子批号 | 处置关闭即可，不生成新子批（简化） |
| 通知方式 | 看板刷新 + 页内消息 |

## 9. 页面与路由规划

### 9.1 路由表

| 路径 | 页面 | 角色 |
|------|------|------|
| `/login` | 登录 | 全部 |
| `/qc` | QC 首页（快捷入口） | QC |
| `/qc/lots` | 生产批列表 / 新建 | QC |
| `/qc/lots/:id` | 生产批详情、新建子批 | QC |
| `/qc/pending` | 待检队列 | QC |
| `/qc/inspect/:subLotId` | 检验录入 | QC |
| `/admin` | 管理看板 | Manager |
| `/admin/holds` | Hold 列表与处置 | Manager |
| `/admin/trace/:lotId` | 追溯详情 | Manager |
| `/admin/settings` | SKU/模板/位置（只读或简易编辑） | Manager |

### 9.2 UI/UX 要求（平板优先）

- 最小点击区域 **44×44 px**；主按钮全宽。  
- 水活输入：**大号数字键盘**组件，支持小数 2 位。  
- 批次码输入：支持手动输入 +「模拟扫码」按钮填入预置码。  
- 顶部 persistent 显示当前子批号与状态颜色（待检=橙、Hold=红、合格=绿）。  
- 管理看板支持 **自动刷新**（建议 3–5 秒轮询）。

## 10. 技术方案建议

### 10.1 推荐栈（与 design guide §8 对齐）

| 层级 | 选型 | Demo 说明 |
|------|------|-----------|
| 前端 | React 18 + Vite + TypeScript | 组件化，便于后续扩展 |
| UI | Tailwind CSS 或 Ant Design Mobile | 快速做平板布局 |
| 后端 | FastAPI (Python) 或 NestJS | REST API |
| 数据库 | SQLite（Demo） | 单文件，部署简单 |
| 部署 | Docker Compose 或 Vercel+Railway | 会议现场需稳定 URL |

> 若团队更熟 .NET，可替换为 ASP.NET Core + SQLite，本计划不限定语言，以 **2 周内可交付** 为准。

### 10.2 架构示意

```text
┌─────────────┐     HTTPS      ┌─────────────┐
│  浏览器/PWA  │ ◄────────────► │  API 服务    │
│  (平板/PC)  │                │  + SQLite   │
└─────────────┘                └─────────────┘
```

### 10.3 API 模块（最小）

- `POST /auth/login`  
- `GET/POST /production-lots`  
- `GET/POST /drying-sub-lots`  
- `GET /pending-inspections`  
- `POST /inspections`（含判定逻辑）  
- `GET /dashboard/summary`  
- `POST /dispositions`  
- `POST /demo/seed`（仅 Demo 环境）

### 10.4 非功能（Demo 级）

| 项 | 要求 |
|----|------|
| 响应时间 | 常规 API &lt; 500ms（局域网） |
| 并发 | 支持 5 人同时演示访问即可 |
| 安全 | Demo 环境 HTTPS；生产前重做 auth |
| 备份 | 会议前导出 SQLite 快照 |

### 13.1 议程（建议 45–60 分钟）

### 13.2 关键台词（摘录）

- 「包装前检验、来料检验可以用同一套模板机制加节点，首期建议仍从烘干后开始。」

## 14. 风险、假设与会上话术

| 风险 | 影响 | 缓解 |
|------|------|------|
| 客户认为 Demo=交付系统 | 期望过高 | 开场与界面「演示数据」标注 |
| 限值与实际不符 | 质疑专业性 | 强调虚构，调研带表单校准 |
| 子批模型与现场不一致 | 需返工 | 会上重点确认 F-11、L-11 |
| 开发周期不足 | 演示不完整 | 砍 P1，保证 P0 两路径 |
| 现场网络差 | 无法访问 | 本地部署备份 |

**核心假设：** 客户认可「先解决信息延时，再解决工艺变异」的分步策略（与设计指南 §1.4.2 一致）。
