## 一、总体判断：Phase 2 不是从零开始

提案把 Phase 2 描述为"新增十个模块（A–J）+ 可选 K"。但从现有代码库看，**其中约一半已经有地基甚至已基本建成**。这份计划书最重要的价值，是先把"已建 vs 待建"分清楚，避免重复开发、避免另起炉灶造成数据孤岛。

现有 `eee-main` 已上线/在建的模块（与 Phase 2 直接相关的）：

| 现有模块 | 路由 | 与 Phase 2 的关系 | 成熟度 |
|---------|------|------------------|--------|
| **Production / Daily Report** | `production` + `/tablet` kiosk | 提案模块 A、B 的主体 | Phase 2 已推进到 M2.1（工单主数据、`prod_run` 单一事实源、平板打卡、逐车成型挂 QC 车、停机事件）|
| **QC（质量控制）** | `qc` | 提案模块 C（烘房）、G（检测放行）的主体 | 成熟：烘干房网格、check-in/out、取样、水活/水份判定、处置、放行、每日报表、追溯 |
| **Warehouse / Inventory** | `warehouse` | 提案模块 H（成品仓 FIFO）的主体，模块 D（领料）的库存底座 | 成熟：只增流水账本、批次生命周期、收货、调拨、放行/拒收、临期、QC↔库存同步（S4）|
| **Packaging（打包）** | `packaging` | 提案模块 D、E、F 的落点 | 仅 Phase 1 MVP：从 QC 放行车 FIFO 出库（`pkg_outbound`），**无领料/班次记录/栈板 QC** |
| **HR（人力资源）** | `hr` | 提案模块 I（考勤绩效请假）的落点 | 员工目录 + 请假/绩效/薪酬/福利页面框架已在（`leave` / `performance` / `payroll` 等），**但考勤未与生产打卡打通** |
| **Auth / Users + MFA** | `auth` | 提案的角色权限 + 受限操作 MFA 的底座 | 成熟：RBAC 权限树、Microsoft Authenticator MFA（`MfaGate`、`reset-user-mfa`）——即提案所说"继承自第一阶段模块 I" |
| **导出框架** | 各模块 | 提案反复提到"复用第一阶段导出框架" | 成熟：`xlsx`/SheetJS 模板导出（QC `TestingExportPage`）、jsPDF |
| **条码/扫码** | QC / Production / Packaging | 提案的扫码入库、扫车贴 | 成熟：`html5-qrcode` 扫码、`jsbarcode`/QR 车贴打印、扫码枪键盘输入 |
| **文档 / SOP** | `docs` / faq | 提案模块 J 的雏形 | 仅静态文档页，**无文件库/版本/审批/自动生成 SOP** |

> **结论**：真正"从零新建"的是 **模块 D/E/F（包装领料+班次+栈板QC）**、**模块 J（文档中心）**、**模块 K（Sage 100，可选）**。其余模块 A/B/C/G/H/I 都是**在现有模块上做增量扩展**。

## 三、横切能力（所有模块共享，先规划避免各自造轮子）

这些是提案在"系统架构 / 核心设计原则"里点名、且**多模块复用**的能力。建议作为 Phase 2 的**公共基础任务**先行或并行拉通：

### 3.1 角色与权限矩阵（RBAC）
- 现状：`permissionStructure.ts` 已有 `finance/hr/qc/warehouse/packaging/production/auth/…` 模块权限树，按 `模块 / resource / action` 三级；MFA 门（`MfaGate`）已存在。
- Phase 2 任务：
  - 新增/补全 Packaging（领料 `pick`、班次 `traveler`、栈板 `pallet`）、Document（`document`）等 resource 的权限项。
  - 落实提案的 5 角色语义映射到现有权限组：**员工 / 主管 / 管理员 / HR / 系统管理员**。现有系统是"权限点"模型而非"固定角色"模型——建议用**权限模板/预设**表达这五个角色（可新增一层"角色预设"或用现有权限批量授予）。
  - 明确哪些 action 属于**受限操作**（见 3.2）。

### 3.2 受限操作 + MFA（继承 Phase 1 模块 I）
- 现状：MFA（Microsoft Authenticator）已在 Auth 模块实现。
- Phase 2 任务：把以下高影响操作接入 MFA 二次校验（`MfaGate` 复用）：
  - **关单**（模块 A/B，含物料核算三选项）
  - **放行授权**（模块 G，产品流转到成品库）
  - 关键主数据/权限变更（系统管理员）。
- 交付：统一的"受限操作前置 MFA"拦截约定（前端门 + RPC 侧校验），列一张"受限操作清单"。

### 3.3 按 SKU 配置字段
- 提案：特定字段（**重量 vs 数量**、必填工单字段、日期码类型）按产品 SKU 预配置，录入时自动生效。
- 现状：`prod_product_master`（生产标准主数据）与 `qc_product_sku`（QC SKU）两套产品主数据并存，经"工单桥"关联（见 12 号文档 M-129）。
- Phase 2 任务：在 SKU 主数据上新增**录入配置字段**（计量口径=重量/数量、日期码类型 Best By/MFG/…、必填项开关）。决策点：**是否合并两套产品主数据**（提案模块 K 会引入 Sage 物料主档，届时更需统一）——建议在本阶段确认归并策略（现有文档已标注"后续如需统一再做归并 migration"）。

### 3.4 半车关联（跨班接续，追溯不断链）
- 现状：生产侧已实现 `prod_run.continues_prev` + `getCarryOverCart`（M1.2b/M-127），逐车成型经 `prod_run.sub_lot_id` 直链 QC 车（M2.1/M-129）。
- Phase 2 任务：把"独立新车 / 半车接续"做成**录入本班第一车时的必选项**（提案明确"必填不可跳过"），并在烘房（模块 C）沿用同一 sub_lot 身份，保证物理车链贯穿 成型→烘干→包装→栈板。

### 3.5 导出框架 / 条码 / i18n
- 复用现有 `xlsx` 模板导出、jsPDF、`html5-qrcode`、车贴打印、三语 i18n。新模块的报表/导出直接套用，不重造。

### 模块 A — 生产·主管批量录入 · 【扩展 Production】

| 状态 | 任务 |
|------|------|

| 🔶 | 主管侧的**半车接续 UI**（当前主要在平板端，管理端列已就绪、交互待补，见 12 号文档 §M1.1 末）|
| 🆕 | **关单（受限操作 + MFA）**：主管/管理员对工单执行关单，三选项——① 无剩余 ② 剩余物料作废（系统按未用量**平均分摊到已完成各车**核算）③ 保留剩余物料。需新增关单记录 + 物料核算逻辑（为模块 K 回推 Sage 做准备）|

### 决策 1 — 产品主数据归并 🔴

**问题**：同一产品被记录在**三张不同的表**、各用不同编号——`prod_product_master`（键 `item_number`，存生产速率/工序）、`qc_product_sku`（键 `code`，存烘干时长/检测限值）、`item`（键为数字 id，库存物料），未来模块 K 还会引入 **Sage 100 物料主档**（第四套）。目前靠"桥"临时对接（QC↔仓库 `qc_sku_item` 关联表；生产↔QC "工单号桥"）。要不要合并？

**建议方案（"中心枢纽"，不做大合并）**：
- **不物理合并**现有三表——风险高，且 Sage 才应是最终"物料主档权威源"。
- **建一张轻量产品交叉引用表（xref）**：用统一 SKU 编码记录"此产品 = prod 哪条 = qc 哪条 = item 哪条 = Sage 哪个料号"。
- **Phase 2 新增的 SKU 配置字段集中放一处**（建议新建 `sku_config` 表挂在统一编码上，而非塞进三张老表任意一张）——见横切能力 3.3。
- 等模块 K 上马、Sage 料号到位后，**以 Sage 料号为权威键**再做正式归并 migration。

**理由**：现有文档已标注"后续如需统一再做归并 migration"。用对照表+集中配置，既能立即支撑"按 SKU 配置字段"，又不承担大合并风险，还为 Sage 预留权威键。

---

### 决策 2 — 一车多工单 🟡

**问题**：提案模块 A 说"一辆车可关联多个一级工单号"，但 `prod_run` 的 `work_order_id` 是单值。要改成多值吗？

**关键澄清**：提案里是两个不同的"单"——**工单号**（生产工单 `prod_work_order`，通常一车一单）与**一级单**（原材料编号，一辆车可消耗多批原料）。"一车多单"指的是后者（原料来源）。

**建议方案（不动生产工单，新建原料关联）**：
- **保持 `prod_run.work_order_id` 单值不变**（生产工单一车一单，清晰）。
- **新建"车 ↔ 一级单(原材料)"多对多关联表**，记录每辆车用了哪些原料批次/一级单。
- 这张表顺带成为模块 K"关单时向 Sage 推送**原料实耗**"的数据底座。

**理由**：把两个维度混进 `work_order_id` 会破坏现有工单累计/车数去重逻辑（BR-P4）。独立关联表既表达"一车多原料"，又服务 Sage 回推与关单核算。

---

### 决策 3 — 箱级 vs 栈板级库存粒度 🟡

**问题**：现有库存追踪到"车/批次"级。模块 H 要"箱级追踪"。若每箱单独记一笔 `inventory_transaction`（只增流水），数据量会暴涨。按箱还是按栈板？

**建议方案（栈板为库存单位，箱为属性）**：
- **以栈板（pallet）作为库存承载单位**——每个栈板一个唯一"栈板牌号"（类似物流 license plate / SSCC）。
- **箱数作为栈板属性字段**记录（模块 F 本就采集"箱数"），而非每箱一条流水。
- 追溯时从栈板下钻看箱数/日期码；**仅当箱子会脱离栈板单独移动/出货**时才拆到箱级。成品一般整栈板流转到最终出货，栈板级足够。

**理由**：避免流水爆炸，同时满足"从栈板到出库"的追溯（模块 H）。栈板牌号即天然的箱级追溯锚点。

---

### 决策 4 — 返烘队列模型 🟢（地基已存在）

**问题**：模块 C（烘房）与 G（检测放行）都提到"不合格批次进返烘队列、重新入烘干"。新建状态机还是复用？

**核对代码发现**：现有 QC 处置类型**已含 `redry_dryer`（返烘）**，子批状态里也有 `awaiting_recheck`（待复检）。地基已在。

**建议方案（复用处置，仅加视图）**：
- **不新建状态机**——返烘就是现有 `redry_dryer` 处置。
- "返烘队列" = 一个**筛选视图**：被判返烘、等待重新入烘干的车。
- 在模块 C 的 Dry Rooms 页加一个"**待返烘**"面板，供重新 check-in。

**理由**：功能已有一半，避免重造状态机与数据不一致。

---

### 决策 5 — 角色表达（5 个角色）🔴

**问题**：提案要 5 个固定角色（员工/主管/管理员/HR/系统管理员）。但现有系统是**权限点模型**——按 `模块/资源/动作` 逐条授权（`user_permission_grant`），**没有"角色"概念**（`erp_user.role` 只是职称文本，不控权限）。怎么套？

**建议方案（加一层"角色预设"）**：
- 新建**权限预设/模板表**：定义 5 个命名预设，每个 = 一组预配权限点。
- 新建用户时**选预设一键套用**作为起点，之后仍可按人微调。
- 受限操作（关单/放行）通过 MFA 门另行控制，与角色正交（见 3.2）。

**理由**：既满足提案"5 角色"表达，又不破坏现有灵活的细粒度权限模型。

---

### 决策 6 — 栈板 QC vs 烘干 QC 🟡

**问题**：模块 F 栈板检测（水份/水活/金属 FE·NFE·SUS）与现有烘干后 QC 检测，同一套还是分开？

**核对代码发现**：现有 `qc_inspection_template` 按 SKU 存检测限值（项目名+上下限），对象是**烘干车**。栈板检测对象是**成品栈板**，且**金属检测是全新的**（带阈值）。

**建议方案（独立表 + 共享字典）**：
- **栈板 QC 建独立表**（对象/字段差异大，硬塞进烘干车模型会别扭）。
- **检测项限值字典可共享**——水份/水活复用现有 `qc_inspection_template` / TestTypesPage 机制。
- **金属检测（FE/NFE/SUS + 阈值）作为新增配置**，可纳入决策 1 的 SKU 配置。

**理由**：结构清晰、复用限值配置，同时容纳金属检测这块全新需求。

### 决策 7 — `prod_operator` ↔ HR 员工映射 🟢（地基已存在）

**问题**：车间 ~159 名一线工存轻量花名册 `prod_operator`（工号→姓名），多数无系统账号。模块 I 考勤/绩效/请假要按"人"统计，需把"工号"对上"HR 员工"。怎么建？

**核对代码发现**：`prod_operator` 表**已有 `erp_user_id` 字段**（可空，指向 HR 员工），关联通道**已存在**，只是没填数据。

**建议方案（无需改表，只需定范围）**：
- **技术上零改动**——映射字段现成。
- 需 NPIC 拍板**范围**：HR 管全部 159 人，还是一部分？
  - 只做整线产出统计 → 用工号即可，不用映射。
  - 要按人算考勤/请假/绩效 → 给这部分人建 HR 员工档案（可无登录的轻量账号），填 `erp_user_id`。
- 建议：**给需要 HR 功能的工人建档并关联**，纯产出统计的继续用工号。

**理由**：关联字段已有，成本仅在数据补录 + 确认范围。

### 决策 8 — Sage 100 连接方式 🔵（需 NPIC IT，可最后做）

**问题**：模块 K 要与 Sage 100 双向同步，技术上怎么连？取决于 NPIC 的 Sage 环境。

**建议方案（走中间库，先验证）**：
- 常见对接：**ODBC 驱动**（Sage 100 的 ProvideX ODBC）、**中间暂存库**、或 **Sage 100 Web Services / 第三方模块**。
- **不直连 Sage 数据库**，走"**中间暂存库 + Supabase Edge Function 拉/推**"，降低耦合与风险。
- **需向 NPIC IT 问清**：① Sage 100 版本；② 是否开放 ODBC/API 读写权限；③ 网络/凭据由谁提供。
- K 为**可选项且排最后**，建议先做小型技术验证（spike）确认可行再投入。

**理由**：外部集成不确定性最高，中间库隔离风险；可选项不阻塞主线。

### 决策 9 — 自动生成 SOP 的范围 🟡

**问题**：模块 J 既要"文件库"又要"自动生成 SOP"，后者不确定性最高。先做哪个？AI 怎么用？

**核对代码发现**：`@google/genai`（Gemini SDK）**已在依赖里**，AI 能力现成可用。

**建议方案（三步走，AI 放最后）**：
1. **v1 文件库**：上传/分类/版本历史/生命周期审批/阅读确认——价值确定、需求清晰。
2. **v2 模板式 SOP 生成**：用**确定性模板**从结构化数据拼装（SKU 配置→包装规程、QC 检测项→检查清单），不涉 AI，可控可审。
3. **v3 AI 辅助起草**：用 `@google/genai` 生成草稿，人工审核后发布。

**理由**：先交付确定价值，AI 生成作为增量，避免一上来押注最不确定的功能。

### 决策 10 — 烘房时间/温度记录 🟡

**问题**：模块 C 提到"时间/温度记录"，提案自己标注"需求确认阶段敲定"，范围未定。

**建议方案（区分人工 vs 传感器）**：
- **需问 NPIC**：要"人工定时抄读温度"还是"传感器/IoT 自动连续记录"？
- **人工抄读** → 小改动，在 check-in/out 加温度录入字段，v1 可做。
- **传感器自动采集** → 硬件集成，范围大，建议**推到 Phase 3**。
- 建议 v1 只做人工抄读，传感器集成明确排除本期。

**理由**：避免把范围不清的硬件集成误纳入本期报价与排期。
